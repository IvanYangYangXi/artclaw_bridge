# 5.9 Plan 模式 — 详细规划文档

> 版本: v1.0 | 日期: 2026-03-28 | 状态: 规划中

---

## 1. 需求分析

### 用户场景
用户希望 AI 在执行复杂任务前先展示计划，用户可以:
- 审阅计划步骤
- 删除不需要的步骤
- 批准后逐步执行
- 中途停止，稍后继续执行剩余步骤

### 核心功能
1. **Plan 生成**: AI 生成结构化的步骤列表
2. **Plan 展示**: UI 中展示带状态的步骤卡片
3. **Plan 编辑**: 删除未开始的步骤
4. **Plan 执行**: 逐步执行，实时更新状态
5. **Plan 暂停/继续**: 中断后可恢复

---

## 2. OpenClaw 兼容性评估

### 2.1 OpenClaw 是否支持 Plan?

**结论: OpenClaw 没有原生 Plan 模式。**

OpenClaw 的通信模型是:
- `chat.send` → AI 回复（可能包含多轮 tool call）→ 最终文本回复
- 每轮对话是原子性的，没有"分步执行"概念
- AI 的 tool call 是自主的，不可从外部干预单步执行

### 2.2 可行方案

**方案: ArtClaw 层实现 Plan 模式**

核心思路: 将 Plan 拆解为多轮 chat.send，每步一轮。

```
用户: "请帮我创建5个材质" (Plan 模式)
    ↓
[ArtClaw 拦截] → 发送给 AI: "请为以下任务制定步骤计划（仅列出步骤，不要执行）: 帮我创建5个材质"
    ↓
AI 返回 Plan (JSON 格式)
    ↓
[UI 展示 Plan] → 用户审阅/编辑 → 用户点击"执行"
    ↓
[ArtClaw 逐步执行] → 每步: chat.send("请执行步骤N: {步骤描述}")
    ↓
每步完成后更新 UI 状态 → 检查是否被停止 → 继续下一步
```

## 3. 技术设计

### 3.1 Plan 数据模型

```cpp
// Plan 步骤状态
enum class EPlanStepStatus : uint8
{
    Pending,    // 待执行
    Running,    // 执行中
    Done,       // 已完成
    Failed,     // 失败
    Skipped,    // 已跳过/已删除
};

// 单个步骤
struct FPlanStep
{
    int32 Index;
    FString Title;           // 步骤标题
    FString Description;     // 详细描述
    EPlanStepStatus Status = EPlanStepStatus::Pending;
    FString Result;          // 执行结果
    FDateTime StartTime;
    FDateTime EndTime;
};

// 完整 Plan
struct FPlan
{
    FString PlanId;          // UUID
    FString UserRequest;     // 原始用户请求
    TArray<FPlanStep> Steps;
    bool bIsExecuting = false;
    bool bIsPaused = false;
    int32 CurrentStepIndex = -1;
};
```

### 3.2 Plan 生命周期

```
[Plan Mode OFF]           [Plan Mode ON]
      |                        |
  用户输入              用户输入 + Plan模式开关开启
      |                        |
  直接 chat.send          ① 拦截用户输入
      |                  ② 发 "制定计划" 提示给 AI
  正常流程               ③ 解析 AI 返回的 Plan JSON
                         ④ UI 展示 Plan 步骤列表
                               |
                      用户操作 (编辑/删除/执行)
                               |
                         ⑤ 逐步 chat.send
                         ⑥ 每步完成 → 更新状态
                         ⑦ 全部完成或被停止
```

### 3.3 Plan 格式约定

AI 返回 Plan 时需要遵循的格式（通过 system prompt 注入）:

```json
{
  "plan": {
    "steps": [
      {"index": 1, "title": "创建基础材质", "description": "创建 M_Base 母材质，设置 Metallic/Roughness 参数"},
      {"index": 2, "title": "创建金属材质", "description": "基于 M_Base 创建 MI_Metal 实例"},
      ...
    ]
  }
}
```

**Plan 解析策略**: 
- 优先匹配 JSON code block (```json ... ```)
- 回退: 正则匹配 numbered list
- 再回退: 纯文本按行拆分

### 3.4 Python 侧实现

```python
# plan_manager.py (新文件)

class PlanManager:
    def __init__(self, bridge):
        self.bridge = bridge
        self.current_plan = None
        self._paused = False
    
    def generate_plan(self, user_request: str) -> dict:
        """让 AI 生成计划（不执行）"""
        prompt = f"请为以下任务制定步骤计划。只输出计划JSON，不要执行任何操作:\n{user_request}"
        response = self.bridge.send_chat(prompt)
        return self._parse_plan(response)
    
    def execute_plan(self, plan: dict, on_step_start, on_step_done):
        """逐步执行计划"""
        for step in plan["steps"]:
            if self._paused:
                break
            if step["status"] == "skipped":
                continue
            on_step_start(step)
            prompt = f"请执行以下步骤:\n步骤 {step['index']}: {step['title']}\n{step['description']}"
            result = self.bridge.send_chat(prompt)
            step["result"] = result
            step["status"] = "done"
            on_step_done(step)
    
    def pause(self):
        self._paused = True
    
    def resume(self):
        self._paused = False
        # 从 current_step 继续执行
    
    def remove_step(self, index: int):
        """删除未开始的步骤"""
        step = self.current_plan["steps"][index]
        if step["status"] == "pending":
            step["status"] = "skipped"
```

### 3.5 C++ UI 设计

**Plan 展示区域**: 在消息列表上方或作为特殊消息类型

```
┌──────────────────────────────────┐
│ 📋 执行计划 (3/5 步完成)          │
│                                  │
│ ✅ 1. 创建基础材质               │
│ ✅ 2. 创建金属材质               │
│ 🔄 3. 创建木材材质 (执行中...)    │
│ ⬜ 4. 创建布料材质         [删除] │
│ ⬜ 5. 设置材质属性         [删除] │
│                                  │
│ [暂停] [取消全部]                 │
└──────────────────────────────────┘
```

**UI 组件**:
- Plan 卡片: 可折叠的 SExpandableArea
- 每步: 状态图标 + 标题 + 删除按钮（仅 Pending 状态）
- 底部: 暂停/继续 + 取消按钮

### 3.6 Plan 模式开关

**方案**: 在输入区域底部工具栏增加 Plan 模式开关

```
[/new] [Create Skill] [管理] [En]  ... [📋 Plan 模式: 关]
```

或在发送时提供选项:
```
[发送] [▼] → 下拉菜单: "直接发送" / "生成计划后执行"
```

## 4. 风险与争议

### 4.1 Plan 格式解析不稳定
**风险**: AI 输出格式不统一，解析失败
**缓解**: 
- 多层回退解析策略
- 如果解析失败，回退为普通对话模式
- 使用强 system prompt 约束输出格式

### 4.2 步骤粒度控制
**争议**: 谁决定步骤拆分? AI 可能拆得太细或太粗
**建议**: 
- v1 由 AI 决定，用户可删除步骤
- v2 考虑让用户自定义步骤

### 4.3 上下文连续性
**风险**: 逐步 chat.send 时，每步是独立对话还是连续对话?
**决定**: 同一个 session 内连续发送，保持上下文

### 4.4 步骤间依赖
**风险**: 前面步骤失败，后面步骤可能也失败
**建议**: 
- 步骤失败时暂停并提示用户
- 用户可选择跳过或重试

### 4.5 与停止按钮的交互
**设计**: 
- "停止"按钮在 Plan 模式下变为"暂停"，只停止当前步骤
- "取消全部"按钮取消整个 Plan

## 5. 分期实施建议

### Phase A: 基础 Plan 模式 (MVP)
- Plan 模式开关 UI
- AI 生成 Plan + JSON 解析
- Plan 列表展示（只读）
- 一键执行所有步骤
- 预计: 4-6h

### Phase B: 交互增强
- 删除未开始的步骤
- 暂停/继续
- 步骤失败处理
- 预计: 3-4h

### Phase C: 高级功能
- Plan 持久化（跨会话恢复）
- 步骤重排序
- 自定义步骤
- 预计: 3-4h

## 7. 待讨论问题

1. **Plan 模式入口**: 开关按钮 vs 发送下拉菜单 vs slash command (`/plan`)? → 建议 slash command `/plan` + 按钮开关双入口
