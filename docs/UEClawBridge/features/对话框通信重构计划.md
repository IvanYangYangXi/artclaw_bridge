# 对话框通信重构开发计划

## 背景

UE 对话框长期存在"发送消息后收不到 AI 回复"的问题，经多次修复仍不稳定。
根本原因是整个通信架构过于复杂，层次太多，每层都引入了新的问题：

- Python 侧：`openclaw_bridge.py` → `openclaw_chat.py` 两层，状态共享混乱
- C++ 侧：字符串拼接传消息、多层文件轮询逻辑交织
- 跨线程信号：asyncio.Event / threading.Event 混用

**决策：移除原有通信代码，按最简原则重新实现。**

---

## 设计原则

1. **一条链路**：用户发消息 → Python 通过 WebSocket 发给 Gateway → 收到回复 → 写文件 → C++ 读文件显示
2. **文件是唯一接口**：C++ 与 Python 之间只通过文件通信，不通过字符串拼接传参
3. **单一入口**：Python 侧只有一个模块 `openclaw_chat.py`，不再拆 bridge/chat 两层
4. **同步等待**：Python 在后台线程同步等待 AI 回复，不用 asyncio，避免跨线程问题
5. **最简轮询**：C++ 每 0.25s 读 stream.jsonl，直到出现 response.txt

## Python 新架构：`openclaw_chat.py`（单文件，≤300 行）

### 通信模型

```
[后台线程 chat_worker]
  1. requests 库 POST /api/chat 或 websockets 同步连接
  2. 循环读 WebSocket 帧，解析 chat 事件
  3. 收到 delta → 追加写 stream.jsonl
  4. 收到 final → 写 response.txt，线程退出

[主线程 / C++ 调用]
  send_chat_async_to_file(msg_file, response_file)
    → 读 msg_file 获取消息内容
    → 启动 chat_worker 线程
    → 立即返回（非阻塞）
```

### WebSocket 连接策略

**每次请求独立 `asyncio.run()`**：不维护全局 asyncio loop，每次 `chat_worker` 线程启动时
调用 `asyncio.run(_do_chat(...))` 完成一次完整的连接→握手→发送→等待回复→断开。

原因：
- 全局共享 asyncio loop 需要跨线程同步，是原来问题的根源
- `asyncio.run()` 在独立线程里是完全安全的
- 代码量减少约 50%，无状态机，无 `_pending` dict，无 `threading.Event`

依赖：`websockets`（UE Python 已内置）

### 模块接口（供 C++ 调用）

```python
# 连接管理
def connect() -> bool
def disconnect() -> None
def is_connected() -> bool

# 聊天（消息通过文件传递）
def send_chat_async_to_file(msg_file: str, response_file: str) -> None

# 会话管理
def reset_session() -> None
def cancel_current_request() -> None

# 诊断（保持兼容）
def diagnose_connection() -> str
```

### 文件协议

| 文件名 | 写入方 | 读取方 | 说明 |
|--------|--------|--------|------|
| `_openclaw_msg_input.txt` | C++ | Python | 用户消息内容（UTF-8） |
| `_openclaw_response_stream.jsonl` | Python | C++ | 实时流式事件，每行一个 JSON |
| `_openclaw_response.txt` | Python | C++ | 完整最终回复，出现即代表结束 |

### stream.jsonl 事件格式

```jsonl
{"type": "delta", "text": "增量文本"}
{"type": "tool_call", "tool_name": "run_ue_python", "tool_id": "xxx", "arguments": {}}
{"type": "tool_result", "tool_id": "xxx", "content": "结果", "is_error": false}
{"type": "final", "text": "完整回复"}
{"type": "error", "text": "[Error] 错误信息"}
```

## C++ 改动：`OpenClawPlatformBridge.cpp`

只改 `SendMessageAsync`，其他方法不动：

```cpp
void FOpenClawPlatformBridge::SendMessageAsync(const FString& Message, const FString& ResponseFile)
{
    // 1. 将消息内容写入临时文件（UTF-8，避免字符串拼接问题）
    FString MsgFile = FPaths::GetPath(ResponseFile) / TEXT("_openclaw_msg_input.txt");
    FFileHelper::SaveStringToFile(Message, *MsgFile,
        FFileHelper::EEncodingOptions::ForceUTF8WithoutBOM);

    // 2. Python 侧读取文件并发送
    FString PythonCmd = FString::Printf(
        TEXT("from openclaw_chat import send_chat_async_to_file\n")
        TEXT("send_chat_async_to_file(r'%s', r'%s')\n"),
        *MsgFile, *ResponseFile
    );
    ExecPython(PythonCmd);
}
```

**注意**：C++ 轮询逻辑（`SendToOpenClaw` 中的 `PollTimerHandle`）**不改**，它本身是正确的。

## 验证方法

在 UE Python 控制台执行：

```python
import openclaw_chat, os, time

# 手动测试
tmp = r"C:/path/to/Saved/UEAgent"
msg_file = tmp + "/_openclaw_msg_input.txt"
resp_file = tmp + "/_openclaw_response.txt"

with open(msg_file, "w", encoding="utf-8") as f:
    f.write("你好，请回复一句话测试")

openclaw_chat.connect()
openclaw_chat.send_chat_async_to_file(msg_file, resp_file)

# 等待 10s 观察文件
for i in range(40):
    time.sleep(0.25)
    stream = tmp + "/_openclaw_response_stream.jsonl"
    if os.path.exists(stream):
        print(open(stream, encoding="utf-8").read()[-200:])
    if os.path.exists(resp_file):
        print("DONE:", open(resp_file, encoding="utf-8").read())
        break
```
