## Phase 6：多平台基础设施

**目标**：UE 插件通用化，移除平台硬编码，支持安装时选择平台

### 6.1 移除平台硬编码 (P0)

**任务**：
- 移除代码中对 `~/.openclaw/` 路径的硬编码
- 将平台相关配置改为从 `~/.artclaw/config.json` 读取
- 统一日志输出，不区分平台

**涉及文件**：
- Python: 各处硬编码 `~/.openclaw/` 的读取逻辑
- 注意：`openclaw_bridge.py` **保持现有命名**，通用逻辑已在 `bridge_core.py`

**验收标准**：
- 代码中不再出现 `~/.openclaw/` 硬编码
- 所有平台相关配置从 `~/.artclaw/config.json` 读取

### 6.2 平台配置持久化 (P0)

**依赖**: 6.1

**任务**：
- 扩展 `~/.artclaw/config.json` 配置结构，新增 platform / skills / mcp 节
- 实现配置的读取和写入（bridge_config.py 扩展）

**配置文件结构**：
```json
{
  "platform": {
    "type": "openclaw",
    "gateway_url": "ws://127.0.0.1:18789",
    "mcp_port": 8080
  },
  "skills": {
    "installed_path": "~/.openclaw/skills",
    "disabled": [],
    "pinned": []
  },
  "mcp": {
    "config_path": "~/.openclaw/openclaw.json",
    "config_key": "mcp.servers"
  },
  "project_root": "D:\\MyProject\\artclaw_bridge"
}
```

**验收标准**：
- 配置文件存在且格式正确
- 重启后配置持久化

### 6.3 首次安装平台选择 (P0)

**依赖**: 6.2

**任务**：
- 检测是否为首次安装（配置文件无 `platform.type`）
- 首次启动时显示平台选择对话框（UE: Slate 弹窗，DCC: Qt 弹窗）
- 选择后保存配置并连接
- **改造 install.bat / install.py**：安装脚本根据平台类型设置默认路径

**UI 流程**：
```
首次启动 → 检测无平台配置 → 显示平台选择对话框
  ├─ OpenClaw  → skills: ~/.openclaw/skills, mcp: ~/.openclaw/openclaw.json
  ├─ WorkBuddy → skills: ~/.workbuddy/skills, mcp: ~/.workbuddy/config.json
  └─ Claude Desktop → skills: ~/.claude/skills, mcp: ~/.claude/config.json
  → 保存配置 → 连接平台
```

**验收标准**：
- 首次安装时弹出平台选择
- 选择后自动保存路径配置并连接
- 非首次安装直接连接
- install.bat 支持平台参数

### 6.4 设置面板扩展 (P1)

**依赖**: 6.2

**任务**：
- 在现有设置弹窗中添加平台配置显示
- 显示当前平台名称、Gateway 地址
- 添加"测试连接"按钮

**验收标准**：
- 设置面板显示当前平台信息
- 测试连接按钮功能正常

### 6.5 重新选择平台 (P1)

**依赖**: 6.4

**任务**：
- 添加"重新选择平台"入口
- 更新配置中的平台类型和相关路径
- 重新连接

**验收标准**：
- 可以切换到其他平台
- 切换后正确更新路径配置并连接新平台

## Phase 7：Skill/MCP 列表展示

**目标**：已安装 Skill/MCP 列表通过配置路径直接扫描获取并展示

### 7.1 Skill 列表扫描 (P0)

**依赖**: 6.2

**任务**：
- 读取 `config.skills.installed_path`
- 扫描目录下所有 Skill 子目录
- 解析 `manifest.json`（代码包）或 `SKILL.md`（文档型）
- 返回列表数据

**验收标准**：
- 正确扫描并返回已安装 Skill 列表
- 同时支持 manifest.json 和纯 SKILL.md 两种格式

### 7.2 MCP Server 列表读取 (P0)

**依赖**: 6.2

**任务**：
- 读取 `config.mcp.config_path`
- 按 `config.mcp.config_key` 定位配置节点
- 解析 MCP Server 配置并返回列表

**验收标准**：
- 正确读取平台配置文件中的 MCP Server 列表

### 7.3 UI 列表展示 (P0)

**依赖**: 7.1, 7.2

**任务**：
- 设置面板中显示 Skill 数量和列表查看入口
- 设置面板中显示 MCP Server 状态
- 点击"查看"展开详细列表

**验收标准**：
- Skill 列表正确显示名称、版本、状态
- MCP Server 列表正确显示连接状态

### 7.4 Skill 同步功能 (P1)

**依赖**: 7.1

**任务**：
- "同步"按钮触发 Skill 同步（复用已有 skill_sync API）
- 同步完成后刷新列表

**验收标准**：
- 同步功能正常工作
- 列表自动刷新

## Phase 8：MCP Server 源码目录整理

**目标**：将 MCP Server 配置模板和文档统一到项目源码中管理，与 Skill 目录结构对齐

### 8.1 创建 mcp_servers 目录 (P0)

**任务**：
- 创建 `mcp_servers/official/` 目录
- 为 ue-editor-agent / maya-editor-agent / max-editor-agent 各创建子目录
- 每个子目录包含 `config_template.json`（各平台配置模板）+ `README.md`
- 创建 `mcp_servers/marketplace/` 目录（预留）

**目录结构**：
```
mcp_servers/
├── official/
│   ├── ue-editor-agent/
│   │   ├── config_template.json
│   │   └── README.md
│   ├── maya-editor-agent/
│   │   ├── config_template.json
│   │   └── README.md
│   └── max-editor-agent/
│       ├── config_template.json
│       └── README.md
└── marketplace/
    └── .gitkeep
```

**验收标准**：
- 目录结构正确
- config_template.json 包含各平台的配置模板
- README.md 包含基本使用说明

## Phase 9：第二平台调研 + POC

**目标**：调研 WorkBuddy（或其他候选平台）的协议兼容性，验证多平台架构可行性

### 9.1 协议调研 (P0)

**任务**：
- 调研 WorkBuddy 的 MCP 协议兼容性
- 调研 WebSocket 接口规范
- 调研配置文件格式和 Skill 目录规范
- 输出调研报告

**验收标准**：
- 输出调研文档，明确协议差异和适配工作量

### 9.2 连接 POC (P0)

**依赖**: 9.1, Phase 6 完成

**任务**：
- 基于调研结果开发最小可用适配器
- 验证消息收发 + MCP 工具调用

**验收标准**：
- 基本消息收发正常
- MCP 工具调用正常

### 9.3 文档更新 (P1)

**任务**：
- 更新架构文档
- 编写新平台接入指南
