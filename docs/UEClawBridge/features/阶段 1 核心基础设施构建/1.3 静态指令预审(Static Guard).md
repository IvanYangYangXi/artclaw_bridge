## 🎯 目标与里程碑

### 核心目标
构建**静态指令预审系统**，在代码执行前通过AST分析拦截危险操作，实现**零运行时风险**的安全沙箱。

### 验收标准
- [ ] 100%拦截`os.system`, `subprocess`等危险调用
- [ ] 敏感文件访问（如`C:\Windows\System32`）被阻止
- [ ] 网络访问必须白名单授权
- [ ] 误报率<5%（合法代码被错误拦截）
- [ ] 预审延迟<50ms（1000行代码）

---

## 📚 背景与上下文

### 为什么需要Static Guard？

在**万能执行器**中，我们允许AI生成的Python代码在UE环境中运行。但如果直接执行未经检查的代码，可能导致：

```python
# ❌ 危险示例：AI可能生成的恶意/误操作代码
import os
os.system("del /f /s /q C:\\*.*")  # 灾难性后果！

import subprocess
subprocess.run(["format", "D:"])  # 格式化硬盘！

# 或者误操作：
import shutil
shutil.rmtree("C:/MyProject/")  # 删除整个项目！
```

### 设计原则

1. **静态优于动态**：在AST层面拦截，不执行代码
2. **白名单优先**：默认拒绝，显式允许
3. **可配置**：支持项目级安全策略
4. **可审计**：记录所有拦截日志

## ❓ 常见问题

### Q1: 如何允许特定的"危险"操作？
A: 在`security_policy.json`中添加`allowed_calls`：
```json
"allowed_calls": ["shutil.rmtree"]
```
或编程方式：
```python
policy = SecurityPolicy()
policy.allowed_calls.add("shutil.rmtree")
```

### Q2: 如何添加自定义的危险模式？
A: 继承`ASTScanner`并重写`visit_Call`：
```python
class CustomScanner(ASTScanner):
    def visit_Call(self, node):
        # 检查自定义危险模式
        if self.get_call_name(node) == "my_dangerous_func":
            self.add_issue(RiskLevel.HIGH, "自定义危险函数", node)
        super().visit_Call(node)
```

### Q3: 性能太慢怎么办？
A: 
- 缓存AST扫描结果（代码哈希）
- 增量扫描（只扫描变更部分）
- 并行扫描（多文件场景）
