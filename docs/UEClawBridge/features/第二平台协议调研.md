## 一、背景

ArtClaw Bridge 当前仅支持 OpenClaw 平台。Phase 9 目标是调研第二平台（Claude Desktop / WorkBuddy）的 MCP 协议兼容性，评估接入可行性和工作量。

## 二、ArtClaw 当前 MCP 架构

```
DCC (UE/Maya/Max)
  └── MCP WebSocket Server (端口 8080/8081/8082)
        ├── 标准 MCP 协议 (JSON-RPC over WebSocket)
        ├── 工具: run_ue_python / run_python (每个 DCC 1 个)
        └── 上下文: get_context (编辑器状态)

OpenClaw Gateway
  └── mcp-bridge 插件
        ├── 连接 WebSocket MCP Server
        ├── 工具自动注册 (tools/list)
        └── 延迟发现 (DCC 后启动)
  └── 聊天链路 (私有 RPC)
        ├── bridge_core.py → openclaw_bridge.py
        └── WebSocket 长连接到 Gateway
```

**关键特征**：
- MCP Server 是 **WebSocket** 传输（非 stdio）
- 聊天是**独立链路**（非 MCP，走 OpenClaw 私有 RPC）
- 两条链路并行：聊天(上行) + 工具调用(下行/MCP)

## 三、Claude Desktop 调研

### 3.1 MCP 支持

- ✅ 原生支持 MCP 协议
- ✅ 支持 **stdio** 传输（主要方式）
- ✅ 支持 **Streamable HTTP** 传输（2025-06-18 新增，替代旧 SSE）
- ❌ **不直接支持 WebSocket** 传输

### 3.2 配置格式

```json
// ~/Library/Application Support/Claude/claude_desktop_config.json (macOS)
// %APPDATA%\Claude\claude_desktop_config.json (Windows)
{
  "mcpServers": {
    "my-server": {
      "command": "python",
      "args": ["path/to/server.py"],
      "env": {}
    }
  }
}
```

### 3.3 接入方案

**方案 A: stdio→WebSocket 桥接器（推荐）**

```
Claude Desktop
  └── stdio MCP Client
        └── artclaw_stdio_bridge.py (stdio 进程)
              └── 连接 DCC 的 WebSocket MCP Server (8080/8081/8082)
                    └── 转发 tools/list, tools/call 等请求
```

实现：
- 写一个 `artclaw_stdio_bridge.py`（~200 行）
- 从 stdin 读 JSON-RPC → 转发到 WebSocket → 从 WebSocket 读响应 → 写到 stdout
- Claude Desktop 配置: `"command": "python", "args": ["artclaw_stdio_bridge.py", "--port", "8080"]`

**方案 B: Streamable HTTP 适配**

- 在 DCC 端同时启动 HTTP MCP 端点
- 工作量大，需要双协议 MCP Server
- 不推荐

### 3.4 聊天链路差异

| 特性 | OpenClaw | Claude Desktop |
|------|----------|----------------|
| 聊天方式 | 独立 WebSocket 长连接 | 无独立聊天链路，用户直接在 Claude Desktop 打字 |
| Bridge 作用 | 双向代理（聊天+工具） | 仅工具调用代理 |
| DCC 内嵌面板 | 有（UE Slate / DCC Qt） | 不需要（用户在 Claude Desktop 界面操作） |
| 流式响应 | 通过 bridge 回传到 DCC 面板 | 直接在 Claude Desktop 显示 |

**结论**: Claude Desktop 只需 MCP 工具链路（下行），不需要聊天链路（上行）。`claude_bridge.py` 比 `openclaw_bridge.py` 简单得多。

### 3.5 工作量评估

| 任务 | 预估 |
|------|------|
| `artclaw_stdio_bridge.py` | ~200 行，2-3 小时 |
| `platforms/claude/` 目录 + 配置模板 | ~1 小时 |
| install.bat/install.py `--platform claude` | 已完成（Phase 6.4） |
| 测试验证 | 需要 Claude Desktop 安装 |

**总计**: ~4 小时（不含测试）

### 3.6 限制

- Claude Desktop 不支持 DCC 内嵌聊天面板（用户必须在 Claude Desktop 窗口操作）
- 无法实现 "DCC 里直接跟 AI 对话" 的体验
- 适合 "Claude Desktop 控制 DCC" 的单向模式

## 四、WorkBuddy 调研

### 4.1 平台概况

WorkBuddy 是企业级 AI Agent 平台，定位类似 OpenClaw，支持多 Agent 协作。

### 4.2 MCP 支持状态

- 需要进一步调研（缺少公开文档）
- 预期支持标准 MCP（stdio/HTTP）
- 可能有私有通信协议（类似 OpenClaw Gateway）

### 4.3 接入策略

- 如果 WorkBuddy 支持 WebSocket MCP → 直接复用现有 MCP Server
- 如果只支持 stdio → 复用 Claude Desktop 的 stdio 桥接器
- 如果有私有协议 → 需要写 `workbuddy_bridge.py`（类似 `openclaw_bridge.py`）

### 4.4 工作量评估

- **最好情况**（支持 WebSocket MCP）: 仅需配置，~2 小时
- **中等情况**（需要 stdio 桥接）: 复用 Claude 方案，~4 小时
- **最坏情况**（私有协议）: 新写 bridge，1-2 天

## 六、POC 实施计划

### Phase 9.2: Claude Desktop stdio 桥接 POC

**目标**: 最小可用的 stdio→WebSocket 桥接，验证 Claude Desktop 能通过 ArtClaw 的 MCP Server 操控 DCC

**文件**:
```
platforms/claude/
├── artclaw_stdio_bridge.py    # stdio→WebSocket 桥接器
├── README.md                  # 使用说明
└── config/
    └── claude-config-snippet.json  # Claude Desktop 配置模板
```

**核心逻辑** (`artclaw_stdio_bridge.py`):
```python
# 伪代码
async def main():
    ws = await connect(f"ws://127.0.0.1:{port}")
    while True:
        line = await read_stdin()
        request = json.loads(line)
        await ws.send(json.dumps(request))
        response = await ws.recv()
        write_stdout(response)
```

**验收标准**:
- Claude Desktop 能看到 ArtClaw 的 MCP 工具（run_ue_python / run_python）
- Claude Desktop 能通过工具执行 DCC 操作
- get_context 返回正确的编辑器状态
