## 1. 现状分析

### 1.1 当前架构

```
memory_core.py (1748行, 共享核心)
├── MemoryManagerV2  
│   ├── 三级模型: short_term(4h) → mid_term(7d) → long_term(永久)
│   ├── 语义标签: fact/preference/convention/operation/crash/pattern/context
│   ├── 自动晋升: TTL 到期时根据 importance + access_count 判断
│   ├── 去重合并: SequenceMatcher 文本相似度
│   └── export_briefing(): 注入到首条消息的上下文
├── memory_store.py (UE 适配层)
└── memory_store.py (DCC 适配层)
```

### 1.2 核心问题

1. **记忆使用率为零** — `~/.artclaw/memory/` 空目录，4 个 agent 从未写入
2. **记忆内容缺乏指导价值** — 没有区分个人/团队记忆，缺乏内容质量标准
3. **缺乏主动读取机制** — briefing 只在首条消息注入，后续不检索
4. **缺乏团队同步机制** — 每个 DCC 实例独立 JSON，用户间不共享
5. **定期整理缺失** — maintain() 只做清理，没有智能提炼

## 2. 设计要点 (v2 细化)

### 2.1 注入内容必须极简

MCP Server 注入到 tool result 的记忆提示必须是**提炼后的极短规则**，不是原始记录。

格式要求:
- 每条规则一行，不超过 80 字符
- 只输出直接相关的规则（关键词匹配后过滤）
- 最多注入 3 条规则
- 无关记忆绝不注入（宁可不注入也不注入噪声）

示例:
```
[Memory] ⚠ Rotator(Pitch,Yaw,Roll) 不是 (Roll,Pitch,Yaw)
[Memory] ⚠ FBX路径含中文会崩溃，先转ASCII
```

### 2.2 写入触发: 只记"付出代价才得到的经验"

**核心原则**: 一次性完成的操作不记录。只有**多次尝试或修改才正确完成的**才提炼规则写入。

两种检测场景:

#### 场景 A: 单次对话内 Agent 自己多次尝试

MCP Server 内部维护 `RetryTracker`:
```
call_1: create_material → 失败 (参数错误)
call_2: create_material → 失败 (节点连接错)  
call_3: create_material → 成功
→ 检测到 retry ≥ 2 → 触发规则提炼
```

#### 场景 B: 用户多轮对话纠正

用户说"不对"/"错了"/"重做" → Agent 重新执行 → 最终成功。
这种情况 MCP Server 看不到（对话在 Gateway 侧），需要:
- 在 SKILL.md 中告诉 AI: "当用户指出错误并经过修正后，主动提炼规则并记录"
- AI 自身判断 "这次经历值得记住"

### 2.3 自动触发时机

| 时机 | 做什么 | 实现位置 |
|------|--------|----------|
| **插件启动** | 加载团队记忆 + 个人记忆到内存 | `init_memory_store()` |
| **首条消息** | briefing 注入（已有） | `_enrich_with_briefing()` |
| **每次工具调用** | 追踪执行结果，更新 retry 计数器 | MCP Server `_handle_tools_call()` |
| **同类操作连续失败 ≥2 次** | 自动查记忆，将匹配规则追加到 tool result | MCP Server 内部 |
| **retry 后成功** | 提取"失败→成功"差异，提炼规则，写入记忆 | MCP Server 内部 |
| **对话结束 / 新对话** | 清空 retry 计数器 | `reset_session()` |
| **定时维护 (30min)** | 已有的 TTL/去重清理 | `maintenance_timer` |
| **每周一次** | 智能整理 — 合并同类 crash、提炼 pattern | `maintenance_timer` 扩展 |

**不需要用户主动操作** — 全部是被动触发，插件启动后就自动工作。

### 2.4 Team Memory 文件结构设计

**需求分析**:
- 多用户提交，需要版本管理（Git）
- Agent 读取要快，token 消耗要少
- 内容是高度提炼的规则，不是原始日志

**方案对比**:

| 方案 | 优点 | 缺点 |
|------|------|------|
| 单一 JSON 文件 | 一次读取全部加载 | 多人同时编辑 Git 冲突严重 |
| 按 tag 分多个 JSON | Git 冲突概率低，可按需加载 | 需要多次 IO |
| **按 tag 分多个 Markdown** | Git diff 友好，人类可读可编辑，AI 直接理解 | 解析稍复杂 |

**推荐: 按 tag 分 Markdown 文件** (类似 knowledge base 的 RULES.md)

```
artclaw_bridge/team_memory/
├── README.md                  ← 格式说明 + 维护指南
├── crash_rules.md             ← 崩溃规则 (P0, 每次必读)
├── gotchas.md                 ← 反直觉陷阱 (P0, 每次必读)
├── conventions.md             ← 项目规范 (P1, 首条消息读)
└── platform_differences.md    ← 跨平台差异 (P1, 首条消息读)
```

**文件格式** (crash_rules.md 示例):
```markdown
# Crash Rules

> 每条规则独立一行，格式: [DCC] 规则内容

- [UE] Rotator 构造参数顺序是 Pitch,Yaw,Roll，不是 Roll,Pitch,Yaw
- [UE] TEXT()/LOCTEXT() 宏不能嵌 UTF-8 emoji，用纯文本或 Unicode 转义
- [Maya] FBX 导入路径含中文会崩溃，先转 ASCII 路径
- [Maya] 中文版有两个安装目录 scripts/ 和 zh_CN/scripts/，都必须同步
- [All] bat 文件必须 CRLF 换行，echo 行禁止 > → -> 字符
```

**为什么 Markdown 比 JSON 更适合**:

1. **Token 效率最高**: 纯文本列表比 JSON 的 `{"key":"...", "value":"...", "tag":"..."}` 结构少约 60% token
2. **Git merge 友好**: 每条规则独立一行，追加新规则不会冲突
3. **人类可直接编辑**: 不需要工具，VSCode/记事本就能改
4. **AI 直接理解**: LLM 对 Markdown 列表的理解比 JSON 数组好
5. **按需加载**: crash_rules.md (P0) 每次必读，conventions.md (P1) 仅首条消息读

**读取策略 (token 最优化)**:

| 文件 | 何时读取 | 注入方式 |
|------|----------|----------|
| crash_rules.md | 每次 briefing | 全文注入 (预计 <500 token) |
| gotchas.md | 每次 briefing | 全文注入 (预计 <500 token) |
| conventions.md | 仅首条消息 | 精简后注入 |
| platform_differences.md | 仅首条消息 | 精简后注入 |

**总 token 开销**: 首条消息 ~1500 token，后续消息 ~800 token (只有 crash_rules + gotchas)。

**多用户提交流程**:
1. Agent 在个人记忆中积累 crash/pattern
2. 智能整理时提炼出团队级规则
3. 写入 `team_memory/*.md` (需要用户确认 Git commit)
4. 其他用户 `git pull` 后自动生效
5. Git merge 冲突几乎不会发生（每人追加不同行）

从用户和 Agent 的视角完整走一遍操作流程:

#### 流程 1: 新用户首次使用

```
  │   ├── 加载个人记忆 (memory_v2.json) — 空的

  ├── 首条消息注入 briefing

  └── Agent 从第一条消息就知道所有团队踩过的坑
```

#### 流程 2: Agent 单次对话内 retry

```
用户 → "创建一个发光材质"

  └── 返回错误: "MaterialExpressionMultiply 连接错误"

Agent → 修改代码，再次调用 run_ue_python (第2次)
  ├── MCP Server: RetryTracker 检测到同类操作 retry_count=2
  ├── 自动查记忆: search("material multiply connect")

  ├── MCP Server: RetryTracker 检测到 "失败→成功" 模式
  ├── 提取差异: 第1次代码 vs 第3次代码
  ├── 自动写入个人记忆: 
  │   key="pattern:material_multiply_connect"
  │   value="MaterialExpressionMultiply 的 A/B 输入必须显式连接，不能留空"
```

```
Agent → 执行 → "完成，已调整 12 个灯光"

用户 → "不对，SpotLight 的亮度单位是 cd 不是 lux，你搞混了"
Agent → (SKILL.md 指导) 检测到用户纠正
  ├── 主动查记忆: search("spotlight brightness unit")
  ├── 修正操作并重新执行
  └── 主动写入记忆:

```

#### 流程 4: 定期智能整理

```
maintenance_timer 触发 (每周)
  ├── 扫描个人记忆中 tag=crash/pattern 的条目
  ├── 合并相似条目 (SequenceMatcher)
  ├── 提炼精简规则 (截断到单行 ≤80 字符)
  ├── 检查是否已存在于 team_memory/
  │   └── 不重复 → 追加到对应 .md 文件末尾
  └── 通知用户: "发现 3 条新的团队级规则，是否提交到 team_memory?"

      └── 用户拒绝 → 保留在个人记忆中
```

#### 流程 5: Agent 反复出错时主动查记忆

```
Agent 连续 2 次执行同类操作失败
  ├── MCP Server: RetryTracker.retry_count >= 2
  ├── 自动搜索记忆 (个人 + 团队)
  ├── 找到相关规则 → 注入到 tool result
  │   "[Memory] ⚠ 相关规则: Rotator(Pitch,Yaw,Roll) 不是 (Roll,Pitch,Yaw)"
  └── Agent 看到提示 → 修正代码 → 成功
  
如果没找到相关记忆:
  ├── 注入: "[Memory] 无相关历史记录，建议仔细检查 API 文档"
  └── Agent 自行排查
```

### 2.6 遗漏点补充

**之前遗漏的交互场景**:

1. **用户主动说"记住这个"** → Agent 直接写入，不需要 retry 检测
2. **用户切换 DCC** (Maya→Max) → 团队记忆自动可用，个人记忆按 DCC 隔离
3. **记忆冲突** → 个人记忆有条目 A，团队记忆有冲突的条目 B → 团队优先
4. **记忆过时** → UE 版本升级后旧规则可能失效 → 规则加 `[UE5.3]` 版本标记
5. **记忆爆炸** → 个人记忆 short_term 积累太多 operation 记录 → 已有容量清理机制
6. **多 Agent 共用记忆** → qi/xiaoli/xiaoshu/xiaoyou 连接不同 DCC 时，个人记忆按 DCC 隔离 (maya/memory_v2.json vs max/memory_v2.json)，团队记忆共享
7. **离线场景** → 团队记忆是本地文件，不依赖网络
8. **记忆可观测性** → 用户怎么知道记忆里有什么？ → Skill 管理面板可以列出/编辑/删除记忆条目 (Phase 4 UI)

## 3. RetryTracker 详细设计

MCP Server 内部组件，追踪工具调用的 retry 模式。

### 3.1 数据结构

```python
class RetryTracker:
    """追踪工具调用的重试模式"""
    
    def __init__(self):
        # 按操作意图分组的调用历史
        # key = operation_fingerprint (工具名 + 操作意图摘要)
        self._history: Dict[str, List[CallRecord]] = {}
        # 最近 N 次调用（用于跨操作分析）
        self._recent_calls: deque = deque(maxlen=50)
    
    @dataclass
    class CallRecord:
        tool_name: str
        code_hash: str       # 代码的语义指纹（去除变量名和空白后的 hash）
        is_error: bool
        error_type: str      # 错误类型的简短分类
        timestamp: float
        code_snippet: str    # 代码前 200 字符
        result_snippet: str  # 结果前 200 字符
```

### 3.2 操作指纹提取

关键问题：怎么判断"同一类操作"？

```python
def _extract_fingerprint(self, tool_name: str, code: str) -> str:
    """提取操作意图指纹
    
    不能用完整代码 hash（每次 retry 代码都不同）。
    需要提取"操作意图"：用了哪些 API、操作了什么类型的对象。
    """
    # 提取 import 语句和关键 API 调用
    api_calls = set()
    for line in code.split('\n'):
        line = line.strip()
        # 提取 unreal.XXX / cmds.XXX 等 API 调用
        for match in re.finditer(r'(?:unreal|cmds|mc|rt)\.\w+', line):
            api_calls.add(match.group())
        # 提取 import 的模块
        if line.startswith(('import ', 'from ')):
            api_calls.add(line.split()[1])
    
    # 排序后 hash → 同类操作的指纹相同
    fingerprint = f"{tool_name}:" + ",".join(sorted(api_calls))
    return hashlib.md5(fingerprint.encode()).hexdigest()[:12]
```

### 3.3 触发逻辑

```python
def on_tool_call(self, tool_name: str, code: str, is_error: bool, 
                 error_msg: str, result: str) -> Optional[str]:
    """记录调用结果，返回需要注入的记忆提示（如果有）
    
    Returns:
        None — 无需注入
        str — 需要追加到 tool result 的记忆提示
    """
    fp = self._extract_fingerprint(tool_name, code)
    
    record = CallRecord(
        tool_name=tool_name,
        code_hash=fp,
        is_error=is_error,
        error_type=self._classify_error(error_msg),
        timestamp=time.time(),
        code_snippet=code[:200],
        result_snippet=result[:200]
    )
    
    self._history.setdefault(fp, []).append(record)
    self._recent_calls.append(record)
    
    history = self._history[fp]
    consecutive_failures = self._count_consecutive_failures(history)
    
    # === 触发 1: 连续失败 ≥2 次 → 自动查记忆 ===
    if is_error and consecutive_failures >= 2:
        hints = self._search_memory_for_hints(code, error_msg)
        if hints:
            return hints  # 注入到 tool result
    
    # === 触发 2: retry 后成功 → 提炼规则写入记忆 ===
    if not is_error and consecutive_failures >= 1:
        # 有之前的失败记录，现在成功了
        self._extract_and_store_lesson(history)
        # 清空该操作的历史
        self._history[fp] = [record]
    
    return None

def _search_memory_for_hints(self, code: str, error_msg: str) -> Optional[str]:
    """从记忆中搜索相关提示"""
    try:
        mm = self._get_memory_manager()
        if not mm:
            return None
        
        # 提取搜索关键词: 错误信息中的关键词 + 代码中的 API 名
        keywords = self._extract_search_keywords(code, error_msg)
        
        all_hints = []
        for kw in keywords[:3]:  # 最多搜索 3 个关键词
            results = mm.search(kw, tag="crash", limit=2)
            results += mm.search(kw, tag="pattern", limit=2)
            for r in results:
                rule = self._format_hint(r)
                if rule and rule not in all_hints:
                    all_hints.append(rule)
        
        if not all_hints:
            return None
        
        # 最多 3 条，每条 ≤80 字符
        hints = all_hints[:3]
        return "\n".join(f"[Memory] ⚠ {h}" for h in hints)
    except Exception:
        return None

def _extract_and_store_lesson(self, history: List[CallRecord]):
    """从 失败→成功 的历史中提炼教训并写入记忆"""
    try:
        mm = self._get_memory_manager()
        if not mm:
            return
        
        failures = [h for h in history if h.is_error]
        success = [h for h in history if not h.is_error][-1]
        
        # 提取教训: 最后一次失败的错误 + 成功的代码差异
        last_failure = failures[-1]
        
        lesson_key = f"pattern:{success.tool_name}:{int(time.time())}"
        lesson_value = {
            "error": last_failure.result_snippet,
            "fix": success.code_snippet,
            "retry_count": len(failures),
            "auto_extracted": True  # 标记为自动提取
        }
        
        mm.record(
            key=lesson_key,
            value=lesson_value,
            tag="pattern",
            importance=min(0.9, 0.6 + len(failures) * 0.1),  # retry 越多越重要
            source="retry_tracker"
        )
    except Exception:
        pass
```

## 6. Token 开销估算

| 场景 | Token 消耗 | 说明 |
|------|-----------|------|
| 首条消息 briefing | ~1500 | crash_rules + gotchas + conventions + 个人 briefing |
