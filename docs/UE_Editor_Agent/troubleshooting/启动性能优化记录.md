# 启动性能优化记录

> 日期：2026-03-17
> 问题：UE 编辑器启动后，MCP Server 需要 80+ 秒才能就绪，严重影响开发体验
> 最终效果：MCP Server 就绪时间从 **80 秒 → 6 秒**

---

## 问题现象

启动 UE 编辑器后，Dashboard 显示 "Connecting..." 长时间无响应，UE Log 中可以看到：
- `init_unreal.py` 被执行了 3 次
- MCP Server 反复 shutdown → restart
- `MCP Server: port 8080 NOT listening` 误报
- 从 Engine Init 完成到 MCP Client 连接成功需要 111 秒

## 根因分析

### 问题1：Python init 被重复执行 3 次

**原因：** UE 的 PythonScriptPlugin 在不同阶段多次 exec `init_unreal.py`：
1. Engine Init 阶段（PythonScriptPlugin 扫描 Content/Python/）
2. Dashboard 构造函数里 `ConnectOpenClawBridge()` → import + 调用 `_start_mcp_gateway()`
3. PythonScriptPlugin 延迟加载阶段再次 exec

每次 exec 都重新定义所有函数和变量，模块级标志被重置，导致 `_initialize()` 和 `_start_mcp_gateway()` 被多次调用。

**关键发现：** UE 的 `ExecPythonCommand` 不走标准 `import` 机制，`sys.modules` 缓存不生效，模块级变量每次都被重新赋值。

### 问题2：Engine Init 阶段 asyncio 无法运转

**原因：** MCP Server 的 asyncio 事件循环通过 `register_slate_post_tick_callback` 驱动。Engine Init 阶段 Slate 还没开始 tick，所以 asyncio 的回调堆积着无法执行，websockets server 的端口绑定永远完不成。

第1次 init 在 Engine Init 阶段就调用了 `_start_mcp_gateway()`，注册了 asyncio bridge 和 verify tick，但都跑不动。

### 问题3：C++ 端重复触发 MCP 启动

**原因：** Dashboard 构造函数中的 `ConnectOpenClawBridge()` 检测到端口没开，就执行：
```python
from init_unreal import _start_mcp_gateway
_start_mcp_gateway()
```
这会 shutdown 已在启动中的 MCP 实例，然后重新创建，导致反复 shutdown→restart 循环。

### 问题4：asyncio tick 每帧只推进 1 步

**原因：** `_UEAsyncBridge._on_tick()` 每次 Slate tick 只调用一次 `stop()` + `run_forever()`，仅执行一批就绪回调。启动阶段编辑器帧率很低（1-2 FPS），每秒只推进 1-2 步，websockets server 的多步启动流程（DNS → socket → bind → listen）需要很多 tick 才能完成。

---

## 修复方案

### 修复1：初始化守卫 — 用 builtins 跨 exec 持久化

**文件：** `Content/Python/init_unreal.py`

```python
import builtins as _builtins

if not getattr(_builtins, '_UE_AGENT_INITIALIZED', False):
    _builtins._UE_AGENT_INITIALIZED = True
    _initialize()
else:
    UELogger.info("Python layer already initialized, skipping duplicate _initialize()")
```

**要点：**
- 不能用模块级变量（每次 exec 会被重置）
- 不能用 `sys.modules`（UE exec 不走 import）
- `builtins` 在进程生命周期内持久，跨 exec 有效

### 修复2：MCP 启动延迟到首个 Slate tick

**文件：** `Content/Python/init_unreal.py`

```python
def _initialize():
    # ... 日志、依赖检测 ...
    if deps_ready:
        # 不直接调用 _start_mcp_gateway()
        # 延迟到首个 Slate tick，确保 asyncio bridge 能运转
        def _deferred_mcp_tick(delta_time):
            try:
                _start_mcp_gateway()
                _register_shutdown_hook()
            finally:
                unreal.unregister_slate_post_tick_callback(_deferred_mcp_handle)
        _deferred_mcp_handle = unreal.register_slate_post_tick_callback(_deferred_mcp_tick)
```

**要点：** Engine Init 阶段 Slate 不 tick，asyncio 跑不动，必须延迟。

### 修复3：`_start_mcp_gateway()` 防重入

**文件：** `Content/Python/init_unreal.py`

```python
def _start_mcp_gateway():
    import builtins as _bi
    if getattr(_bi, '_UE_MCP_GATEWAY_STARTING', False):
        UELogger.info("MCP Gateway startup already in progress, skipping")
        return
    _bi._UE_MCP_GATEWAY_STARTING = True
    # ... 启动逻辑 ...
```

### 修复4：C++ 端不再重复触发 MCP 启动

**文件：** `UEAgentDashboard.cpp`

Dashboard 构造函数里的 `ConnectOpenClawBridge()`：
- **Before：** 检测端口没开 → `_start_mcp_gateway()` 重启
- **After：** 只检查状态 + 打日志，MCP 启动完全由 `init_unreal.py` 的延迟 tick 负责

Dashboard 构造函数里的 MCP 状态检查：
- **Before：** 构造时立即检查，误报 `port 8080 NOT listening`
- **After：** 延迟 3 秒再检查（`FTSTicker` 3.0f delay）

### 修复5：asyncio tick 每帧多跑 10 轮

**文件：** `Content/Python/mcp_server.py`

```python
def _on_tick(self, delta_time: float) -> None:
    if self._loop is None or self._loop.is_closed():
        return
    try:
        for _ in range(10):
            self._loop.stop()
            self._loop.run_forever()
    except Exception as e:
        UELogger.error(f"Async tick error: {e}")
```

**要点：** 每次 Slate tick 推进 10 轮 asyncio，启动阶段即使 1-2 FPS 也能每秒推进 10-20 步。

|------|--------|--------|------|
| `_initialize()` 执行次数 | 3 次 | 1 次 | ✅ 守卫生效 |
| MCP Gateway 启动次数 | 3 次 (循环重启) | 1 次 | ✅ 无重复 |

## 经验教训

1. **UE PythonScriptPlugin 的 exec 行为**：不走标准 import，模块级状态不可靠。跨 exec 状态必须用 `builtins` 或全局字典。
2. **Engine Init 阶段的限制**：Slate 未 tick、asyncio 无法运转、端口检测必然失败。所有依赖异步的操作必须延迟到首个 tick。
3. **asyncio 在 UE 中的驱动效率**：每 tick 只跑 1 轮远远不够，启动阶段帧率低时尤为明显。多轮推进是关键优化。
