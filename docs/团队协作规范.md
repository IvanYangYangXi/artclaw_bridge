## 2. Git 工作流程

### 2.1 分支策略
- **main**: 主分支，稳定版本
- **develop**: 开发分支，日常开发
- **feature/***: 功能分支，从 develop 分支创建
- **bugfix/***: bug 修复分支，从 develop 或 main 分支创建
- **release/***: 发布分支，从 develop 分支创建

### 2.2 提交规范
#### 提交信息格式
```
<type>(<scope>): <subject>

<body>

<footer>
```

#### Type 类型
- **feat**: 新功能
- **fix**: 修复 bug
- **docs**: 文档更新
- **style**: 代码格式（不影响功能）
- **refactor**: 重构
- **perf**: 性能优化
- **test**: 测试相关
- **chore**: 构建过程或辅助工具的变动

#### 示例
```
feat(ui): add chat window component

- Implement basic chat UI
- Add message list widget
- Integrate input field

Closes #123
```

### 2.3 工作流程
1. **开发新功能**:
   ```bash
   git checkout develop
   git pull origin develop
   git checkout -b feature/your-feature-name
   # 开发代码
   git add .
   git commit -m "feat: add your feature"
   git push origin feature/your-feature-name
   # 创建 PR 到 develop 分支
   ```

2. **修复 bug**:
   ```bash
   git checkout develop
   git pull origin develop
   git checkout -b bugfix/your-bugfix-name
   # 修复代码
   git add .
   git commit -m "fix: resolve the bug"
   git push origin bugfix/your-bugfix-name
   # 创建 PR 到 develop 分支
   ```

3. **合并 PR**:
   - 需要 Code Review
   - 通过所有测试
   - 至少一人 approve
   - 使用 Squash and merge 合并

## 3. 代码规范

### 3.1 命名规范
- **文件名**: 使用中文命名（文档），英文命名（代码文件）
- **类名**: PascalCase (例如: `FClawPlugin`)
- **函数名**: PascalCase (例如: `ProcessData`)
- **变量名**: CamelCase (例如: `inputData`)
- **常量名**: SCREAMING_SNAKE_CASE (例如: `MAX_BUFFER_SIZE`)
- **宏定义**: SCREAMING_SNAKE_CASE (例如: `CLAW_API`)

### 3.2 注释规范
```cpp
/**
 * 函数功能描述
 *
 * @param ParamName 参数描述
 * @return 返回值描述
 */
TSharedPtr<FClawData> ProcessData(const TSharedPtr<FClawData>& Input);
```

### 3.3 代码格式
- 缩进：4 空格
- 大括号：K&R 风格
- 行宽：最大 120 字符
- 使用 clang-format 格式化代码

## 5. 任务管理

### 5.1 Issue 创建
- 标题清晰，描述具体
- 包含任务类型（feature/bug/enhancement）
- 添加合适的标签
- 分配给对应责任人
- 设置优先级和里程碑

### 5.2 任务状态
- **Open**: 待处理
- **In Progress**: 进行中
- **Review**: 审查中
- **Done**: 已完成

### 5.3 提交要求
- 每位成员完成各自任务后，需要及时提交代码
- 提交前确保代码编译通过
- 提交前运行测试
- 提交信息必须符合规范

## 6. 沟通机制

### 6.1 定期会议
- **每日站会**: 15 分钟，同步进展和问题
- **每周评审**: 1 小时，评审成果和计划调整
- **里程碑会议**: 每个阶段结束，总结和规划

### 6.2 沟通工具
- **即时通讯**: Teams/Slack
- **文档协作**: Confluence/Google Docs
- **项目管理**: GitHub Issues/Projects
- **版本控制**: Git

### 6.3 汇报规范
- 完成任务后及时向团队汇报
- 遇到阻塞问题立即通知
- 每日更新任务进度
- 重要决策需要团队讨论

## 7. 质量保证

### 7.1 代码审查
- 所有代码需要经过 Code Review
- 至少一人 approve 才能合并
- Review 时重点关注：
  - 代码质量和规范性
  - 性能和安全性
  - 测试覆盖率
  - 文档完整性

### 7.2 测试要求
- 单元测试覆盖率 > 80%
- 集成测试必须通过
- 性能测试达标
- 内存无泄漏

### 7.3 发布流程
1. 创建 release 分支
2. 完成测试和文档更新
3. 合并到 main 分支
4. 创建 GitHub Release
5. 打包分发

## 8. 安全规范

### 8.1 敏感信息
- 不要提交 API 密钥、密码等敏感信息
- 使用环境变量或配置文件管理敏感信息
- 提交前检查代码中的敏感信息

### 8.2 权限管理
- GitHub 仓库权限按角色分配
- 重要的合并操作需要管理员权限
- 定期审查访问权限

## 9. 紧急情况处理

### 9.1 发现重大 bug
1. 立即创建 issue，标记为 critical
2. 通知团队成员
3. 创建 hotfix 分支
4. 快速修复并测试
5. 合并到 main 和 develop
6. 发布 hotfix 版本

### 9.2 服务中断
1. 立即通知所有相关人员
2. 快速定位问题
3. 临时恢复服务
4. 彻底修复问题
5. 总结经验教训

## 10. 持续改进

### 10.1 定期回顾
- 每月进行团队回顾
- 总结做得好的地方
- 分析需要改进的地方
- 制定改进计划

### 10.2 工具和流程优化
- 根据团队反馈优化协作工具
- 改进开发流程
- 提升开发效率
