## 二、DCC 插件层

### 2.1 插件框架（Skill开发者无需关注）

框架自动处理：MCP注册、参数校验、命令入队/出队、返回值序列化、异常格式化、日志记录。

**主线程安全机制**（UE等DCC的核心约束）：
```
MCP线程 → 入队 → CommandQueue（线程安全）→ 主线程Tick消费 → DCC API → 结果回传
```

**DCC插件通用结构**：
```
{DCC}Agent/
├── bootstrap.py        # 生命周期、MCP服务器启动
├── command_queue.py    # 线程安全命令队列
├── context.py          # DCC上下文采集
└── skills/             # Skill集合（开发者主战场）
    ├── scene/
    ├── asset/
    ├── material/
    └── ...
```

### 2.2 Skill开发规范

**一个Skill只需做三件事**：声明元数据 + 定义参数Schema + 写DCC API逻辑。

```python
from artclaw import skill, SkillResult

@skill(
    name="create_static_mesh",
    description="在场景中创建静态网格体",
    category="mesh",
    params={
        "mesh_path": {"type": "string", "required": True},
        "location": {"type": "vec3", "default": [0,0,0]},
    }
)
def create_static_mesh(mesh_path: str, location=(0,0,0)) -> SkillResult:
    # 只需关注DCC API，框架处理其余一切
    actor = unreal.EditorLevelLibrary.spawn_actor_from_object(
        unreal.load_asset(mesh_path), unreal.Vector(*location)
    )
    return SkillResult.success({"actor_name": actor.get_name()})
```

**Skill分类与优先级**：

| 级别 | 说明 | 优先级 |
|------|------|--------|
| P0 核心 | 覆盖90%日常操作 | 最高 |
| P1 常用 | 专业工作流常规操作 | 高 |
| P2 高级 | 特定场景专业能力 | 按需 |

### 2.3 UEEditorAgent 特殊设计

UE采用C++ + Python混合，C++负责生命周期/UI/主线程调度，Python负责MCP通信和Skill逻辑：

```
Source/ (C++)
├── UEEditorAgentModule     # 模块定义
├── EditorAgentSubsystem    # 生命周期管理
├── SlateUI/                # 控制面板
└── CommandBridge/          # 主线程调度桥

Content/Python/
├── mcp_server.py           # MCP服务器
├── command_queue.py        # 命令队列
└── skills/                 # Skill实现（主要工作）
```

## 三、桥接通信层（openclaw-mcp-bridge）

职责：提供可复用的MCP服务器基础设施，各DCC插件的Python端共享。

```
openclaw-mcp-bridge/
└── openclaw/
    ├── server.py           # MCPServer核心
    ├── skill_registry.py   # Skill注册与分发
    ├── transport/          # stdio / SSE / WebSocket
    └── protocol/           # MCP类型、消息处理、参数校验
```

| 场景 | 传输方式 |
|------|---------|
| 本地AI客户端 | stdio（性能最优） |
| 局域网/远程 | WebSocket |

## 四、多平台管理层

专门解决**多DCC × 多AI平台**管理复杂度。

**统一配置**（artclaw-config.yaml）：
```yaml
platforms:
  workbuddy:
    endpoint: "ws://localhost:9090"
    auth: { type: bearer_token, token_env: WORKBUDDY_TOKEN }

dcc_instances:
  ue_editor:
    software: unreal_engine
    mcp_port: 7001
    auto_start: true
```

**关键机制**：

| 组件 | 职责 |
|------|------|
| 会话路由器 | 按实例名/软件类型将请求路由到正确DCC端口 |
| 健康监控 | 30s心跳检测，失败3次标记UNHEALTHY，自动重连 |
| 认证适配器 | ApiKey/BearerToken/OAuth2，对DCC层透明 |
| 配置热更新 | 文件监听，新增/删除平台无需重启 |

**工具命名空间**（多实例并存时的隔离方式）：
```
ue_editor.create_actor
ue_editor.get_scene_info
maya_primary.export_fbx
```

## 七、CLI 调试模式（Debug Mode）【待开发 / Future】

> **状态**：🚧 待开发（Backlog）
> **优先级**：低 — 当前阶段专注 MCP 链路，CLI 作为未来扩展预留设计

为提升可调试性和开发体验，系统预留 CLI 调试模式作为 MCP 协议的补充。CLI 模式直接与 UE Python 环境交互，绕过 MCP 的 JSON-RPC 封装，降低调试复杂度。

**注意**：CLI 模式是**独立于 MCP 的开发者工具**，不是 Agent 的替代方案。Agent 始终通过 MCP 与 UE 交互。

### 7.1 架构位置（预留设计）

```
┌──────────────────────────────────────────────┐
│  开发者 / AI Agent                            │
└──────────┬─────────────────┬─────────────────┘
           │                 │
     ┌─────▼─────┐    ┌──────▼──────┐
     │  MCP模式  │    │  CLI调试模式 │
     │(当前重点) │    │  (待开发)   │
     └─────┬─────┘    └──────┬──────┘
           │                 │
           └────────┬────────┘
                    │

     │  ┌──────────────────────┐  │
     │  │  CLI Parser          │  │  ← 预留接口

     │           │                │
```

### 7.2 CLI 调试模式核心特性（设计草案）

| 特性 | 说明 | 适用场景 |
|------|------|----------|
| **--dry-run** | 显示实际执行的 Python 代码，不真正执行 | 本地调试 Skill |
| **--show-python** | 输出 Skill 对应的完整 Python 源码 | 代码审查 |
| **管道支持** | `artclaw skill pipe "cmd1 | cmd2 | cmd3"` | 批量处理脚本 |
| **本地执行** | 直接调用 UE Python，无网络/JSON-RPC 开销 | 离线环境 |
| **REPL 模式** | `artclaw shell` 进入交互式 Python 环境 | 快速验证 API |
| **无头模式** | `artclaw run --headless` | CI/CD 自动化 |

### 7.3 CLI 与 MCP 的关系

> **当前阶段**：Agent 始终通过 **MCP 模式** 与 UE 交互。CLI 模式为**预留扩展**，供未来开发者直接使用。

```
当前阶段（Phase 0~4）：
├── 用户/Agent → MCP 模式 → UE Editor Agent
└── CLI 模式：待开发

未来阶段（Backlog）：
├── 开发者/脚本 → CLI 模式 → Skill Hub
└── Agent/OpenClaw → MCP 模式 → Skill Hub
```

**设计原则**：
- **MCP 模式**：Agent 与 UE 交互的**唯一通道**（当前重点）
- **CLI 模式**：开发者工具，用于**本地调试、CI/CD、离线环境**（未来扩展）

### 7.4 CLI 命令规范

```bash
# 基础命令
artclaw skill list                    # 列出所有 Skill
artclaw skill info <name>             # 查看 Skill 详情
artclaw skill test <name> [args]      # 本地测试 Skill

# 调试模式
artclaw skill test create_static_mesh --mesh_path="/Game/Cube" --dry-run
# 输出：
# [DRY-RUN] 将执行以下 Python 代码：
# actor = unreal.EditorLevelLibrary.spawn_actor_from_object(...)
# return {"actor_name": actor.get_name()}

# 管道模式（阶段 4 实现）
artclaw skill pipe "get_selected_actors | filter_static_mesh | set_material_red"

# REPL 模式
artclaw shell
>>> S = unreal.EditorLevelLibrary.get_selected_level_actors()
>>> print([a.get_name() for a in S])

# 无头模式（CI/CD）
artclaw run --headless --script="batch_optimize_lights.py"
```

### 7.5 实现要点

1. **CLI Parser 模块**：基于 `argparse`，解析命令并路由到 Skill Hub
2. **管道机制**：参考 Unix pipe，前一个 Skill 的 stdout 作为后一个的 stdin
3. **与 MCP 共享执行层**：CLI 和 MCP 最终都调用相同的 Skill Hub，确保行为一致
4. **权限继承**：CLI 直接继承操作系统权限，审计通过 shell history
