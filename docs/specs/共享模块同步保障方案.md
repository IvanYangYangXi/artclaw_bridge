# 共享模块同步保障方案

> **版本**: v1.0  
> **日期**: 2026-03-31  
> **状态**: 执行中

---

## 一、问题概述

ArtClaw Bridge 采用"唯一源码 + 安装时复制"的共享模块管理模式：

```
core/                    ← 唯一源码
platforms/openclaw/      ← 唯一源码（平台特定）
    ↓ install.py 复制
subprojects/UEDAgentProj/.../Content/Python/   ← 副本
subprojects/DCCClawBridge/core/                ← 副本
```

**问题**：开发过程中直接修改了副本（UE 或 DCC 端）而没有同步回源码，导致三端漂移。2026-03-31 审查发现 4 个文件不一致。

## 二、已完成的修复

### 2.1 漂移修复

| 文件 | 漂移方向 | 修复方式 |
|------|---------|---------|
| bridge_core.py | core 比 DCC 新 | core → DCC |
| memory_core.py | core 比 DCC 新 | core → DCC |
| openclaw_chat.py | UE 比 platform 新（pinned_hint 功能） | UE → platform |
| openclaw_ws.py | UE 比 platform 新（异常处理修正） | UE → platform |

### 2.2 校验工具 verify_sync.py

项目根新增 `verify_sync.py`，对比所有共享模块的 MD5 hash：

```bash
python verify_sync.py              # 检查模式
python verify_sync.py --fix        # 源码 → 副本
python verify_sync.py --reverse    # 副本 → 源码
python verify_sync.py --ci         # CI 模式（不一致退出码 1）
```

覆盖的文件对：
- `core/` 的 6 个模块 → DCC core/ + UE Content/Python/（共 12 对）
- `platforms/openclaw/` 的 3 个模块 → UE Content/Python/（共 3 对）

### 2.3 mcp_configs/ 改名

`mcp_servers/` → `mcp_configs/`，明确只存配置模板不含 MCP Server 代码。

## 三、同步保障机制

### 3.1 开发流程规范

**源码优先原则**：

1. 修改共享模块时，**必须先改源码**（`core/` 或 `platforms/`）
2. 然后运行 `python verify_sync.py --fix` 同步到副本
3. 如果在 UE/DCC 端调试时直接改了副本，事后用 `--reverse` 回写源码

**异常发现路径**：

- 开发者提交前运行 `verify_sync.py`
- CI 在 PR 检查中运行 `verify_sync.py --ci`
- install.py 安装后提示运行校验

### 3.2 install.py 集成

install.py 安装结束后自动调用 verify_sync，确认源码与刚部署的副本一致。

### 3.3 Git pre-commit hook（可选）

```bash
# .git/hooks/pre-commit
python verify_sync.py --ci || {
    echo "共享模块不一致！运行 python verify_sync.py --fix"
    exit 1
}
```

## 四、P1 MCP Server 协议核心提取（降级为 P2 优化项）

### 4.1 现状

UE 和 DCC 各有独立的 `mcp_server.py`，功能相同但驱动方式不同：

| | UE 版 (32KB, 935行) | DCC 版 (22KB, 649行) |
|-|---------------------|---------------------|
| 事件循环 | Slate tick 驱动 asyncio | 独立线程 asyncio |
| JSON-RPC | 完全相同 | 完全相同 |
| MCP 握手 | 完全相同 | 完全相同 |
| Resources | ✅ 支持 | ❌ 无 |
| 工具注册 | Phase 模块化注册 | 内联注册 run_python |
| 状态同步 | C++ Subsystem + bridge 状态文件 | 无 |

### 4.2 评估

- **重复代码**：~300 行（JSON-RPC 编解码 + MCP 握手 + 工具路由）
- **改动面**：两个生产代码都要重写
- **收益**：有限（只有 2 个文件受影响）
- **风险**：UE 版有大量 UE 特定逻辑，提取后需要仔细测试

**决定**：降级为 P2 优化项。当前优先保障 P0 同步机制。未来如果要做独立 MCP Server（standalone 模式），再提取。
