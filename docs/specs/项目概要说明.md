# ArtClaw 项目概要说明

## 一、项目定位

**ArtClaw** 是一个桥接框架，让各类DCC软件（UE、Maya、Max等）通过MCP协议接入AI Agent平台（WorkBuddy、OpenClaw等），获得AI操作能力。

**当前阶段**：基于 UE + WorkBuddy 完成第一套完整实现，打通整条链路，形成可复用基建。其他DCC/平台按此模式扩展。

---

## 二、工作量分布

```
┌─────────────────────────────────────────────────────┐
│  Skill 开发（主要工作量，持续投入）                    │
│  针对每个DCC软件逐个实现AI可调用的操作能力              │
│  ⚠️ 这是项目价值的核心，也是长期主战场                 │
├─────────────────────────────────────────────────────┤
│  ArtClaw 框架（一次性建设）                           │
│  通信层、插件框架、多平台管理                          │
│  ✅ 框架目标：Skill开发者只需关注DCC API，其余透明      │
└─────────────────────────────────────────────────────┘
```

---

## 三、项目组成

```
ArtClaw
├── DCC插件层              各DCC软件本地插件（C++/Python）
│   ├── UEClawBridge      ✅ 当前重点
│   ├── MayaAgent          📋 V2.0
│   └── MaxAgent           📋 V2.0
├── 桥接通信层             openclaw-mcp-bridge（Python）
│   ├── MCP服务器（per DCC实例）
│   ├── Skill注册/路由
│   └── 传输层（stdio/WebSocket）
└── 多平台管理层           Platform Manager
    ├── 平台注册与认证适配
    ├── 会话路由
    └── 健康监控/重连
```

---

## 四、关键概念

| 概念 | 说明 |
|------|------|
| **Skill** | AI可调用的单个DCC操作能力，是项目核心价值单元。软件专属，数量庞大（单软件数百个），持续扩展 |
| **MCP** | Model Context Protocol，AI平台与DCC工具之间的标准通信协议 |
| **DCC插件** | 运行在DCC软件内的本地插件，内嵌MCP服务器，管理Skill的注册与执行 |
| **Platform Manager** | 管理多DCC实例 × 多AI平台的连接、路由和监控 |

---

## 五、核心挑战

| 挑战 | 策略 |
|------|------|
| **Skill数量规模大** | 框架承担所有通信/注册复杂度；提供模板，Skill开发只需写DCC API逻辑 |
| **各DCC Python环境冲突** | 每个插件独立虚拟环境，dependency_manager统一管理 |
| **多平台协议差异** | 适配器模式，DCC插件无需感知平台差异 |
| **DCC主线程限制** | 线程安全命令队列，主线程Tick消费，结果异步回传 |
| **多软件多平台管理复杂** | 统一配置文件、会话路由器、健康监控 |
