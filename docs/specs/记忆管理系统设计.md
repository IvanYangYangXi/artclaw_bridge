## 一、设计背景

### 1.1 现状问题

当前 UE 和 DCC 的记忆系统各自独立实现，存在以下不足：

| 问题 | 影响 |
|------|------|
| 无记忆生命周期管理 | facts/preferences/conventions 永久累积，文件膨胀无上限 |
| 无记忆压缩机制 | 重复或近似的记忆无法合并，浪费 AI 上下文窗口 |
| 无崩溃/成功链路记录 | AI 重复踩坑——执行过导致崩溃的操作不会被记住 |
| 两端实现不一致 | UE 三层（facts/preferences/conventions），DCC 三层（system/project/short_term），语义重叠但不对齐 |
| 短期记忆直接过期删除 | 有价值的操作经验随 TTL 丢失，无法沉淀为长期知识 |

### 1.2 设计目标

1. **短/中/长期三级记忆**：按时效和重要性分层，支持自动晋升和压缩
2. **记忆合并去重**：相似 key 自动检测合并，控制文件大小
3. **崩溃/成功链路记忆**：AI 操作结果（成功/失败/崩溃）自动沉淀，提升后续成功率
4. **不直接过期删除**：短期记忆到期后压缩归档到中期，而非丢弃
5. **不跨 DCC 共享**：每个 DCC 维护独立记忆（运行时隔离），但存储格式统一
6. **文件大小可控**：设置各层上限，触发时自动压缩/淘汰

## 二、三级记忆模型

### 2.1 分层概览

```
┌─────────────────────────────────────────────────────────┐
│                    记忆管理器 (MemoryManager)            │
│                                                          │
│  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐  │
│  │   短期记忆    │  │   中期记忆    │  │   长期记忆    │  │
│  │  Short-Term  │──►  Mid-Term    │──►  Long-Term   │  │
│  │              │  │              │  │              │  │
│  │ TTL: 4小时   │  │ TTL: 7天     │  │ 永久(可淘汰) │  │
│  │ 上限: 200条  │  │ 上限: 500条  │  │ 上限: 1000条 │  │
│  │              │  │              │  │              │  │
│  │ 原始操作记录  │  │ 压缩后的经验  │  │ 核心知识     │  │
│  │ 即时上下文   │  │ 模式与规律   │  │ 稳定偏好     │  │
│  │ 临时变量     │  │ 崩溃/成功链路 │  │ 项目规范     │  │
│  └──────────────┘  └──────────────┘  └──────────────┘  │
│                                                          │
│  ┌───────────────────────────────────────────────────┐  │
│  │             操作链路记录 (OperationLedger)          │  │
│  │  崩溃记录 · 成功链路 · 风险操作标记                  │  │
│  └───────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────┘
```

### 2.2 各层详细定义

#### 短期记忆 (Short-Term)

| 属性 | 值 |
|------|-----|
| **用途** | 当前会话或近期操作的原始记录 |
| **TTL** | 4 小时（可配置） |
| **容量上限** | 200 条 |
| **存储内容** | 选中对象缓存、最近执行的命令、临时计算结果、对话上下文片段 |
| **到期行为** | 压缩晋升到中期记忆（不删除） |
| **检索优先级** | 最高（最新最相关） |

#### 中期记忆 (Mid-Term)

| 属性 | 值 |
|------|-----|
| **用途** | 从短期记忆压缩而来的经验总结 |
| **TTL** | 7 天（可配置） |
| **容量上限** | 500 条 |
| **存储内容** | 操作模式（"用户习惯在材质上先调 roughness 再调 metallic"）、崩溃/成功链路摘要、近期偏好趋势 |
| **到期行为** | 高频访问/高重要性条目晋升到长期；低价值条目淘汰 |
| **检索优先级** | 中等 |

#### 长期记忆 (Long-Term)

| 属性 | 值 |
|------|-----|
| **用途** | 经过验证的核心知识 |
| **TTL** | 永久（但受容量上限约束） |
| **容量上限** | 1000 条 |
| **存储内容** | 项目事实（资产路径规范、命名约定）、稳定偏好（灯光参数、材质风格）、确认有效的操作规范、多次验证的崩溃规避规则 |
| **淘汰策略** | 按最后访问时间 + 重要性评分排序，达上限时淘汰底部 |
| **检索优先级** | 基础（始终可用） |

### 2.3 语义分类标签 (tag)

每条记忆除了所属层级，还携带一个语义标签，用于跨层检索和压缩归类：

| 标签 | 含义 | 示例 |
|------|------|------|
| `fact` | 项目事实 | "项目使用 UE 5.4，主关卡路径 /Game/Maps/Main" |
| `preference` | 用户偏好 | "灯光默认强度 5000 lux" |
| `convention` | 项目规范 | "材质命名格式 M_{Type}_{Name}" |
| `operation` | 操作链路记录 | "批量替换材质成功，50 个 Actor，耗时 3s" |
| `crash` | 崩溃/错误记录 | "在 PIE 模式下调用 EditorLevelLibrary 导致崩溃" |
| `pattern` | 行为模式 | "用户总是先选中对象再请求修改材质" |
| `context` | 临时上下文 | "当前正在处理 LightingLevel 关卡" |

## 三、记忆生命周期

### 3.1 记忆写入

```
AI 操作 / 用户交互
    │
    ▼
MemoryManager.record(key, value, tag, importance=0.5)
    │
    ▼
写入短期记忆
    │  附加元数据:
    │    created_at: 时间戳
    │    access_count: 0
    │    last_accessed: 时间戳
    │    importance: 0.5 (0~1)
    │    tag: "operation"
    │    source: "mcp_tool:run_ue_python"
    ▼
短期记忆存储
```

### 3.2 自动晋升（短期 → 中期）

**触发条件**：短期记忆 TTL 到期时，由压缩器批量处理

**压缩规则**：
1. **高频访问**（access_count ≥ 3）→ 直接晋升，保留原文
2. **操作链路类**（tag=operation/crash）→ 合并同类操作为摘要后晋升
3. **上下文类**（tag=context）→ 丢弃（纯临时数据，不晋升）
4. **其他** → 按 importance 评分决定：≥ 0.6 晋升，< 0.6 丢弃

**合并压缩示例**：
```
短期（3 条独立记录）:
  - "执行 set_material on Actor_1 成功"
  - "执行 set_material on Actor_2 成功"  
  - "执行 set_material on Actor_3 成功"

压缩后（中期 1 条）:
  - "批量 set_material 操作: 3次成功, 0次失败 (Actor_1, Actor_2, Actor_3)"
```

### 3.3 自动晋升（中期 → 长期）

**触发条件**：中期记忆 TTL 到期时，或中期容量达上限时

**晋升标准**（满足任一即晋升）：
- access_count ≥ 5（高频使用的知识）
- tag = crash 且 importance ≥ 0.7（重要的崩溃规避规则）
- tag = convention 或 fact 且被多次验证
- importance ≥ 0.8（被显式标记为重要）

### 3.4 合并去重

**触发时机**：每次晋升批处理时 + 定期维护（每 24 小时一次）

**去重策略**：
1. **精确匹配**：key 完全相同 → 保留最新值，累加 access_count
2. **前缀匹配**：key 前缀相同（如 `crash:PIE_mode:*`）→ 合并为一条摘要
3. **语义相似**：同 tag 下 value 文本相似度 > 0.8 → 合并（基于简单文本比较，不依赖向量模型）

### 3.5 生命周期状态图

```
                    ┌─────────┐
                    │  写入    │
                    └────┬────┘
                         │
                         ▼
                 ┌───────────────┐
                 │   短期记忆     │ TTL: 4h
                 │  (Short-Term) │ 上限: 200
                 └───────┬───────┘
                         │ TTL 到期
                         │
              ┌──────────┼──────────┐
              │          │          │
         高频/重要    操作链路     低价值/临时
              │          │          │
              ▼          ▼          ▼
          直接晋升    压缩合并     丢弃
              │          │     (tag=context
              │          │      且 importance<0.6)
              ▼          ▼
                 ┌───────────────┐
                 │   中期记忆     │ TTL: 7d
                 │  (Mid-Term)   │ 上限: 500
                 └───────┬───────┘
                         │ TTL 到期 / 容量满
                         │
              ┌──────────┼──────────┐
              │          │          │
         高频/关键    普通经验     低价值
              │          │          │
              ▼          ▼          ▼
          晋升长期   合并已有      淘汰
              │      长期条目
              │          │
              ▼          ▼
                 ┌───────────────┐
                 │   长期记忆     │ 永久
                 │  (Long-Term)  │ 上限: 1000
                 └───────────────┘
                         │ 容量满
                         ▼
                 LRU + 重要性排序淘汰
```

## 四、操作链路记录 (Operation Ledger)

### 4.1 设计目的

记录 AI 通过 MCP 工具执行的操作结果（成功/失败/崩溃），形成可检索的经验库。
**核心价值**：AI 在执行类似操作前，先查阅历史链路，规避已知崩溃路径，优先选用验证过的成功链路。

### 4.2 链路记录格式

```json
{
  "key": "op:run_ue_python:set_material:2026-03-19T09:30",
  "tag": "operation",
  "value": {
    "tool": "run_ue_python",
    "action": "set_material",
    "params_summary": "target=StaticMeshActor_5, material=/Game/Materials/M_Chrome",
    "result": "success",
    "duration_ms": 120,
    "side_effects": ["material_slot_0_changed"],
    "notes": ""
  },
  "importance": 0.5
}
```

**崩溃记录格式**：

```json
{
  "key": "crash:run_ue_python:PIE_EditorLevelLibrary:2026-03-19",
  "tag": "crash",
  "value": {
    "tool": "run_ue_python",
    "action": "spawn_actor",
    "params_summary": "在 PIE 运行时调用 EditorLevelLibrary.spawn_actor_from_object",
    "result": "crash",
    "error": "UE Fatal Error: Attempting to modify world during PIE",
    "root_cause": "EditorLevelLibrary 不能在 PIE 模式下调用",
    "avoidance_rule": "执行编辑器操作前先检查 is_playing_in_editor()，若为 True 则拒绝执行并提示用户退出 PIE",
    "severity": "critical"
  },
  "importance": 0.95
}
```

### 4.3 自动记录机制

在 MCP 工具执行管道中自动插入记录逻辑：

```
MCP Tool 调用
    │
    ▼
执行前: 查询 Operation Ledger（有无相关崩溃记录?）
    │
    ├── 有崩溃记录 → 注入警告到执行上下文
    │                 "⚠️ 历史崩溃: {avoidance_rule}"
    │
    ▼
执行操作
    │
    ▼
执行后: 记录结果
    ├── 成功 → record(tag="operation", importance=0.5)
    ├── 失败 → record(tag="operation", importance=0.7, result="error")
    └── 崩溃 → record(tag="crash", importance=0.9+)
```

### 4.4 AI 查询接口

通过现有 `memory` MCP 工具的 `search` action 查询：

```
memory(action="search", query="PIE spawn_actor")
→ 返回相关的崩溃记录和成功链路
```

新增专用 action `check_operation`（可选，优化查询效率）：

```
memory(action="check_operation", tool="run_ue_python", action_hint="spawn_actor")
→ 返回该操作的历史成功率、已知风险、建议
```

### 5.3 文件大小控制

| 层级 | 单条大小限制 | 总条目上限 | 预估文件大小上限 |
|------|------------|-----------|----------------|
| short_term | 2 KB | 200 | ~400 KB |
| mid_term | 4 KB | 500 | ~2 MB |
| long_term | 4 KB | 1000 | ~4 MB |
| **合计** | — | 1700 | **~6.4 MB** |

当任一层级达到上限时，触发该层的清理：
- **short_term**：强制批量压缩晋升（即使未到 TTL）
- **mid_term**：低 importance + 低 access_count 的条目淘汰
- **long_term**：LRU 淘汰（last_accessed 最旧 + importance 最低的先走）

## 六、AI 记忆检索策略

### 6.1 核心问题

AI Agent 如何获取记忆？两种极端方案各有弊端：

| 方案 | 优点 | 缺点 |
|------|------|------|
| **全量注入**（每次对话都把所有记忆塞进 system prompt） | AI 总能看到全部信息 | 记忆越积越多 → token 消耗爆炸，6MB 记忆 ≈ 150万 token |
| **纯按需查询**（AI 自己决定何时调用 memory tool） | token 最省 | AI 不知道有什么记忆 → 经常"忘了查"，崩溃规避规则形同虚设 |

### 6.2 混合策略：摘要注入 + 按需检索

采用 **"小摘要 + 精准查"** 的两层设计：

```
┌─────────────────────────────────────────────────────────┐
│                    每次对话开始时                         │
│                                                          │
│  System Prompt 自动注入（固定 token 开销）：              │
│  ┌────────────────────────────────────────────────────┐  │
│  │  📋 记忆摘要（Memory Briefing）                     │  │
│  │                                                     │  │
│  │  ⚠️ 崩溃规避规则（长期, tag=crash, 全量）          │  │
│  │  📐 项目规范（长期, tag=convention, 全量）          │  │
│  │  📌 项目事实（长期, tag=fact, 前 N 条高频）         │  │
│  │  🎨 用户偏好（长期, tag=preference, 前 N 条高频）   │  │
│  │                                                     │  │
│  │  预估大小: 500~2000 token（可配置上限）              │  │
│  └────────────────────────────────────────────────────┘  │
│                                                          │
│  NOT included（不注入，按需查询）：                       │
│  ├── 短期记忆（操作记录、临时上下文）                     │
│  ├── 中期记忆（经验摘要、操作链路）                       │
│  └── 长期记忆中低频/低 importance 条目                   │
│                                                          │
└─────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────┐
│                    对话过程中                             │
│                                                          │
│  AI 通过 MCP Tool 按需检索：                             │
│                                                          │
│  memory(action="search", query="材质替换")               │
│  → 跨层搜索，返回相关条目                                │
│                                                          │
│  memory(action="check_operation", tool="run_ue_python",  │
│         action_hint="delete_actor")                      │
│  → 返回该操作的历史成功率、崩溃记录、建议                 │
│                                                          │
│  memory(action="get", key="crash:PIE_EditorLevelLibrary")│
│  → 精确读取某条记忆                                      │
│                                                          │
└─────────────────────────────────────────────────────────┘
```

### 6.3 摘要注入的内容和预算

#### 自动注入内容（Memory Briefing）

按优先级排列，在 token 预算内尽量多装：

| 优先级 | 内容 | tag | 来源层 | 注入策略 |
|--------|------|-----|--------|---------|
| **P0** | 崩溃规避规则 | `crash` | 长期 | **全量注入**（安全第一） |
| **P1** | 项目规范 | `convention` | 长期 | 全量注入 |
| **P2** | 项目事实 | `fact` | 长期 | 按 access_count 降序取前 N 条 |
| **P3** | 用户偏好 | `preference` | 长期 | 按 access_count 降序取前 N 条 |
| **P4** | 高重要性操作经验 | `operation` | 中期 | importance ≥ 0.8 的取前 5 条 |

#### Token 预算控制

```json
{
  "memory_briefing": {
    "max_tokens": 1500,
    "max_entries_per_tag": {
      "crash": -1,
      "convention": -1,
      "fact": 20,
      "preference": 15,
      "operation": 5
    },
    "format": "compact"
  }
}
```

- `max_tokens`: 摘要总 token 上限（默认 1500，可调整）
- `crash` 和 `convention` 设为 -1 表示不限条数（安全和规范必须全量）
- 超出预算时按优先级从低到高裁剪

#### 紧凑格式（减少 token）

摘要输出采用紧凑格式，而非完整 JSON：

```
[Memory Briefing]
⚠ CRASH RULES:
- EditorLevelLibrary: 禁止在 PIE 模式下调用，先检查 is_playing_in_editor()
- TEXT() 宏: 不能嵌入 UTF-8 emoji，用 Unicode 转义

📐 CONVENTIONS:
- 材质命名: M_{Type}_{Name}
- Actor 标签: 必须含项目前缀

📌 FACTS:
- UE 5.4 | 主关卡 /Game/Maps/Main
- 默认灯光强度 5000 lux

🔧 RECENT OPS:
- 批量材质替换: 94% 成功率，注意资产路径
```

预估 token：~200-800 token（视记忆量而定），远低于全量注入。

### 6.4 自动检索触发（辅助 AI 主动查询）

除了 AI 自己决定何时调用 `memory(action="search")`，系统在以下场景**自动触发**检索并注入工具调用上下文：

| 触发场景 | 检索内容 | 注入方式 |
|---------|---------|---------|
| `run_ue_python` / `run_python` 被调用前 | 查询 `tag=crash` 中与代码关键词匹配的规避规则 | 在工具 description 或返回值中追加警告 |
| 用户消息包含"之前"/"上次"/"历史" | 查询中/短期记忆中的操作记录 | AI 工具调用 prompt 提示 |
| 连续失败 ≥ 2 次 | 查询相关 crash/operation 记录 | 在错误返回中追加历史经验 |

### 6.5 与现有 MCP Resource 的关系

当前 UE 侧的 MCP Server 支持 `resources/read` 协议，可以将 Memory Briefing 注册为一个 Resource：

```
Resource URI: unreal://memory/briefing
Name: Memory Briefing
Description: Key crash rules, conventions, facts, and preferences
```

但 **OpenClaw 当前不主动读取 MCP Resource 注入 prompt**（mcp-bridge 只转发 tool call），所以实际注入路径需要在 bridge 层面实现：

**推荐方案**：在 `bridge_core.py` 或 `openclaw_bridge.py` 的 `send_chat` 流程中，将 Memory Briefing 作为**消息前缀**附加到用户消息之前：

```python
async def send_chat(self, message: str):
    # 获取记忆摘要
    briefing = memory_store.export_briefing(max_tokens=1500)
    
    # 作为系统上下文附加（不改变用户消息本身）
    if briefing:
        enriched_message = f"[Memory Briefing]\n{briefing}\n\n[User Message]\n{message}"
    else:
        enriched_message = message
    
    await self._send_rpc("chat.send", {"message": enriched_message})
```

> **注意**：这种方式会让 AI 每次对话都看到摘要，但 token 开销可控（~200-800 token）。如果 OpenClaw 未来支持 system prompt 注入或 MCP Resource 自动读取，可以迁移到更优雅的实现。

| **`maintain`** | 手动触发维护（压缩/晋升/去重） | **新增**，一般由定时器自动触发 |

## 八、定时维护流程

### 8.1 维护触发时机

| 触发条件 | 执行内容 |
|---------|---------|
| 每 30 分钟（MCP Server tick） | 短期记忆 TTL 检查 → 到期条目批量压缩晋升 |
| 每 24 小时 | 全量维护：中期晋升检查 + 跨层去重 + 文件大小检查 |
| 容量达上限时 | 立即触发对应层级的清理 |
| `memory(action="maintain")` 调用时 | 手动触发全量维护 |

### 8.2 维护流程

```
定时维护触发
    │
    ▼
1. 短期 TTL 扫描
    │  到期条目 → 按压缩规则处理
    │  高频/重要 → 晋升中期
    │  操作链路 → 合并后晋升中期
    │  低价值临时 → 丢弃
    │
    ▼
2. 中期 TTL 扫描
    │  到期条目 → 按晋升标准处理
    │  高频/关键 → 晋升长期
    │  普通经验 → 尝试合并到已有长期条目
    │  低价值 → 淘汰
    │
    ▼
3. 跨层去重
    │  精确 key 匹配 → 合并
    │  前缀匹配 → 合并
    │  相似文本 → 合并（同 tag 内）
    │
    ▼
4. 容量检查
    │  超限层级 → 强制淘汰最低价值条目
    │
    ▼
5. 更新 _meta.stats
    │
    ▼
6. 持久化到磁盘
```
