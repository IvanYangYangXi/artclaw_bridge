```
│               DCC 原生 API / SDK                  │
└──────────────────────────────────────────────────┘
```

**关键区分**：
- **通信框架代码共享**：`bridge_core.py`、`mcp_server.py` 骨架等代码文件被所有 DCC 复用
- **运行时实例独立**：每个 DCC 进程内运行自己的 MCP Server（端口不同）、自己的 Bridge 连接
- **每个 DCC 只注册 1 个 MCP 工具 `run_python`**：其余功能以 Python API 形式提供，AI 通过 `run_python` 调用
- **OpenClaw Skill 按需加载**：`~/.openclaw/skills/` 下的 SKILL.md 按描述匹配，命中时指导 AI 如何通过 `run_python` + Python API 完成任务
- AI 通过工具名前缀区分目标 DCC：`mcp_maya-primary_run_python` vs `mcp_ue-editor_run_ue_python`

## 三、核心模块设计

### 3.1 DCC Adapter 抽象接口

```python
# adapters/base_adapter.py

from abc import ABC, abstractmethod
from typing import Any, Dict, List, Optional, Callable

class BaseDCCAdapter(ABC):
    """DCC 适配层抽象接口，每个 DCC 实现一个子类"""

    # ── 基础信息 ──

    @abstractmethod
    def get_software_name(self) -> str:
        """返回软件名称，如 'maya', 'max'"""

    @abstractmethod
    def get_software_version(self) -> str:
        """返回软件版本，如 '2023', '2024'"""

    @abstractmethod
    def get_python_version(self) -> str:
        """返回内置 Python 版本"""

    # ── 生命周期 ──

    @abstractmethod
    def on_startup(self) -> None:
        """DCC 启动时调用：注册菜单、shelf、启动面板"""

    @abstractmethod
    def on_shutdown(self) -> None:
        """DCC 关闭时调用：清理资源、断开连接"""

    # ── 主线程调度 ──

    @abstractmethod
    def execute_on_main_thread(self, fn: Callable, *args) -> Any:
        """在 DCC 主线程执行函数（场景 API 安全）"""

    @abstractmethod
    def execute_deferred(self, fn: Callable, *args) -> None:
        """延迟到主线程空闲时执行（非阻塞）"""

    # ── 上下文采集 ──

    @abstractmethod
    def get_selected_objects(self) -> List[Dict]:
        """获取当前选中对象列表"""

    @abstractmethod
    def get_scene_info(self) -> Dict:
        """获取当前场景基本信息（场景名、对象数、帧范围等）"""

    @abstractmethod
    def get_current_file(self) -> Optional[str]:
        """获取当前文件路径"""

    # ── UI 集成 ──

    @abstractmethod
    def get_main_window(self) -> Any:
        """获取 DCC 主窗口句柄（作为 Qt 窗口 parent）"""

    @abstractmethod
    def register_menu(self, menu_name: str, callback: Callable) -> None:
        """在 DCC 菜单栏注册入口"""

    # ── 代码执行 ──

    @abstractmethod
    def execute_code(self, code: str, context: Dict = None) -> Dict:
        """在 DCC 环境中执行 Python 代码（万能执行器）"""
```

### 3.2 Maya Adapter 实现要点

```python
# adapters/maya_adapter.py （核心骨架）

import maya.cmds as cmds
import maya.utils
import maya.OpenMayaUI as omui
from shiboken2 import wrapInstance
from PySide2 import QtWidgets

class MayaAdapter(BaseDCCAdapter):

    def get_software_name(self) -> str:
        return "maya"

    def get_software_version(self) -> str:
        return cmds.about(version=True)  # "2023"

    def execute_on_main_thread(self, fn, *args):
        return maya.utils.executeInMainThreadWithResult(fn, *args)

    def execute_deferred(self, fn, *args):
        maya.utils.executeDeferred(fn, *args)

    def get_selected_objects(self):
        sel = cmds.ls(selection=True, long=True)
        result = []
        for obj in (sel or []):
            result.append({
                "name": obj.split("|")[-1],
                "long_name": obj,
                "type": cmds.objectType(obj),
            })
        return result

    def get_main_window(self):
        ptr = omui.MQtUtil.mainWindow()
        return wrapInstance(int(ptr), QtWidgets.QMainWindow)

    def register_menu(self, menu_name, callback):
        if cmds.menu(menu_name, exists=True):
            cmds.deleteUI(menu_name)
        cmds.menu(menu_name, label="ArtClaw", parent="MayaWindow")
        cmds.menuItem(label="Open Chat Panel", command=lambda _: callback())

    def execute_code(self, code, context=None):
        # 在主线程安全执行，注入上下文变量
        ...
```

### 3.3 bridge_dcc.py — DCC 通信适配层

**已在 G3.1 完成**（`core/bridge_dcc.py`）。

与 UE 的 `openclaw_bridge.py` 对应，但使用 Qt signal/slot 替代文件轮询：

```python
from bridge_core import OpenClawBridge  # 平台无关核心（来自 openclaw-mcp-bridge/）

class DCCBridgeManager:
    """DCC 通用的 OpenClaw Bridge 管理器"""

    # Qt signals:
    #   connection_changed(bool, str)  — 连接状态变更
    #   ai_message(str, str)           — AI 流式文本 (state, text)
    #   ai_thinking(str, str)          — AI thinking 文本
    #   response_complete(str)         — 最终响应完成

    def connect(self) -> bool: ...
    def disconnect(self): ...
    def send_message(self, message: str): ...
    def cancel(self): ...
```

**会话管理（Session Key 与新对话）**：

```
Session Key 格式: agent/{client_id}:{timestamp}
  例: agent/maya-editor:1710842400
      agent/max-editor:1710842500
      agent/ue-editor:1710842600
```

- **Session Key 带时间戳**：每次新对话生成唯一 key（格式 `agent/{client_id}:{timestamp}`），Gateway 依此创建独立 session
- **新对话流程**：`reset_session()` → ① 清空 `session_key`（确保下次发送时生成新 key）② 通过 `send_message("/new")` 正常发送（走常规消息路径，AI 回复可流式显示在面板）
- **displayName 动态化**：`bridge_core.py` 内置 `_DISPLAY_NAME_MAP`，根据 `client_id` 自动映射（`maya-editor` → "Maya Claw Bridge"，`max-editor` → "Max Claw Bridge"，`ue-editor` → "UE Claw Bridge"）
- **DCC 上下文注入**：首条消息自动前缀 `[DCC Context]`（含软件名/版本/场景信息），后续消息不重复，新对话时重置

> ⚠️ **重要约束**：不能通过 bridge RPC 直接发送 `/new` slash command——Gateway 收到 `/new` 会关闭 WebSocket 连接（正常关闭 code 1000），导致需要重连。正确做法是先清空 session_key，再以普通消息方式发送 `/new`。

**Session 隔离**：

| DCC | client_id | MCP 端口 | Session 示例 |
|-----|-----------|---------|-------------|
| UE | `ue-editor` | 8080 | `agent/ue-editor:1710842400` |
| Maya | `maya-editor` | 8081 | `agent/maya-editor:1710842500` |
| Max | `max-editor` | 8082 | `agent/max-editor:1710842600` |

**与 UE 适配层的对比**：

| 方面 | UE (openclaw_bridge.py) | DCC (bridge_dcc.py) |
|------|------------------------|---------------------|
| 数据回传 | 文件轮询（C++ FTSTicker 读 JSON 文件） | Qt signal/slot（直接推送到 UI） |
| 日志 | `unreal.log` → UE Output Log | `logging` → DCC Script Editor |
| 线程模型 | 独立 asyncio 线程（相同） | 独立 asyncio 线程（相同） |
| 状态同步 | `_bridge_status.json` 文件 | Qt signal `connection_changed` |
| 依赖 | `import unreal` | `from PySide2.QtCore import Signal` |

**两者都调用同一个 `bridge_core.py`**，只是外壳不同。

### 3.4 MCP Server — 每个 DCC 独立运行

每个 DCC 进程内运行自己的 MCP Server 实例，**只注册 1 个 MCP 工具 `run_python`**，其余功能以 Python API 形式在 `run_python` 内调用：

```
Maya 进程内:                          Max 进程内:
┌──────────────────────┐             ┌──────────────────────┐
│ MCP Server :8081     │             │ MCP Server :8082     │
│                      │             │                      │
│ MCP 工具:            │             │ MCP 工具:            │
│  run_python (唯一)   │             │  run_python (唯一)   │
│                      │             │                      │
│ Python API (非 MCP): │             │ Python API (非 MCP): │
│  adapter.get_selected│             │  adapter.get_selected│
│  adapter.get_scene   │             │  adapter.get_scene   │
│  knowledge_base.*    │             │  knowledge_base.*    │
│  memory_store.*      │             │  memory_store.*      │
│  skill_hub.*         │             │  skill_hub.*         │
│  ...                 │             │  ...                 │
└──────────┬───────────┘             └──────────┬───────────┘
           │ WebSocket                          │ WebSocket
           │                                    │
    ┌──────▼────────────────────────────────────▼──────┐
    │              OpenClaw Gateway                     │
    │  mcp-bridge 插件:                                │
    │    maya-primary → ws://127.0.0.1:8081            │
    │    max-primary  → ws://127.0.0.1:8082            │
    │                                                   │
    │  AI 看到的工具名:                                 │
    │    mcp_maya-primary_run_python                    │
    │    mcp_max-primary_run_python                     │
    └──────────────────────────────────────────────────┘
```

**关键点**：
- **每个 DCC 只暴露 `run_python` 一个 MCP 工具**：AI 通过前缀 `mcp_maya-primary_` / `mcp_max-primary_` 区分目标 DCC
- 原有 `get_selected_objects`、`get_scene_info`、`knowledge_search`、`memory`、`skill_list` 等功能不再注册为 MCP 工具，而是以 Python API 形式存在，AI 通过 `run_python` 执行 Python 代码调用
- MCP Server 代码骨架复用（JSON-RPC 协议、工具注册机制），工具注册极简化
- 端口自动探测（8081+），与 UE 的 8080 错开

### 3.5 主线程命令队列

与 UEClawBridge 相同的模式：MCP/WebSocket 线程不直接调用 DCC API，而是通过命令队列转发到主线程。

```
WebSocket 线程 (MCP Server)       主线程（Qt 事件循环）
     │                                │
     │  command_queue.put(cmd)         │
     │ ──────────────────────────────► │
     │                                │  QTimer(0) 触发消费
     │                                │  adapter.execute_on_main_thread(cmd)
     │  result_future.set_result()    │
     │ ◄────────────────────────────── │
```

**Maya 特有**：Maya 的 Qt 事件循环已经是主线程，可直接用 `QTimer.singleShot(0, fn)` 或 `maya.utils.executeDeferred()` 实现。

## 四、MCP 工具设计

> **v1.3 重构**：每个 DCC 只保留 **1 个 MCP 工具 `run_python`**，其余全部降级为 Python API，AI 通过 `run_python` 执行 Python 代码调用。原有工具功能迁移至 OpenClaw Skill 按需加载（见 §4.4）。

### 4.1 唯一 MCP 工具：`run_python`

| 工具名 | 说明 | 类型 | 备注 |
|--------|------|------|------|
| `run_python` | 万能执行器（在 DCC 中执行 Python） | 写操作 | 各 DCC 的执行环境不同 |

**设计理念**：`run_python` 是 DCC 的唯一入口。AI 通过它执行任意 Python 代码，包括调用内部 Python API 来实现原有 MCP 工具的全部功能。这大幅减少了 MCP 工具注册数量，降低了 AI 的工具选择复杂度。

### 4.2 原 MCP 工具降级为 Python API

以下工具不再注册为 MCP 工具，改为通过 `run_python` 调用对应的 Python API：

#### 通用功能（所有 DCC）

| 原 MCP 工具 | 新状态 | Python API 调用方式 |
|-------------|--------|-------------------|
| `get_editor_context` | → Python API | `adapter.get_editor_context()` |
| `get_selected_objects` | → Python API | `adapter.get_selected_objects()` |
| `get_scene_info` | → Python API | `adapter.get_scene_info()` |
| `knowledge_search` | → Python API | `knowledge_base.search(query)` |
| `memory` | → Python API | `memory_store.get/set/search/list/delete(...)` |
| `skill_list` | → Python API | `skill_hub.list_skills()` |
| `skill_manage` | → Python API | `skill_hub.manage_skill(action, ...)` |

#### Maya 特有功能

| 原 MCP 工具 | 新状态 | Python API 调用方式 |
|-------------|--------|-------------------|
| `get_node_info` | → Python API | `adapter.get_node_info(node_name)` |
| `get_material_info` | → Python API | `adapter.get_material_info(material_name)` |
| `get_animation_info` | → Python API | `adapter.get_animation_info(node_name)` |

> **示例**：AI 需要获取选中对象时，不再调用 `mcp_maya-primary_get_selected_objects` MCP 工具，而是调用 `mcp_maya-primary_run_python` 执行 `adapter.get_selected_objects()`。

### 4.3 工具命名空间

多 DCC 实例并存时，mcp-bridge 插件自动添加 server 名称前缀。每个 DCC 只暴露一个工具：

```
AI 看到的工具名                        实际 MCP 工具名
─────────────────────────────────      ─────────────────
mcp_maya-primary_run_python         →  run_python (Maya :8081)
mcp_max-primary_run_python          →  run_python (Max :8082)
mcp_ue-editor_run_ue_python         →  run_ue_python (UE :8080)
```

### 4.4 OpenClaw Skill 按需加载

原有 MCP 工具的丰富功能并未丢失，而是迁移到 **OpenClaw Skill** 体系（`~/.openclaw/skills/` 下的 SKILL.md）。

#### 工作原理

```
用户请求 → OpenClaw Agent
              │
              ▼
        扫描 available_skills 的 description
              │
              ├── 无匹配 → 直接用 run_python 处理
              │
              └── 命中 Skill → 读取对应 SKILL.md
                                │
                                ▼
                          SKILL.md 指导 AI:
                          "通过 run_python 调用
                           adapter.get_selected_objects()
                           来获取选中对象..."
                                │
                                ▼
                          AI 调用 run_python
                          执行 Python API 代码
```

**核心机制**：
- **按描述匹配**：OpenClaw Agent 启动时扫描所有 Skill 的 `<description>`，只有用户请求匹配时才读取 SKILL.md——不用不占 token
- **SKILL.md 即指南**：每个 SKILL.md 描述该功能的使用场景、可用的 Python API、调用示例，指导 AI 通过 `run_python` 完成任务
- **不注册 MCP 工具**：ArtClaw Skill 运行时不再将 Skill 注册为 MCP 工具，彻底解耦

#### 迁移后的 OpenClaw Skill 清单

| OpenClaw Skill | 对应原 MCP 工具 | 描述 |
|---------------|----------------|------|
| `artclaw-context` | `get_editor_context`, `get_selected_objects`, `get_scene_info` | DCC 编辑器上下文采集（软件信息、选中对象、场景概览） |
| `artclaw-memory` | `memory` | 分层记忆管理（get/set/search/list/delete） |
| `artclaw-knowledge` | `knowledge_search` | 本地知识库检索（API 文档、最佳实践） |
| `artclaw-skill-manage` | `skill_list`, `skill_manage` | DCC 内部 Skill 浏览与管理 |
| `artclaw-highlight` | _(新增)_ | 场景内对象高亮标注 |

## 五、数据流

### 5.1 用户发送消息（完整链路）

```
用户在 Chat Panel 输入
    │
    ▼
chat_panel.py 判断是否 / 命令
    │
    ├── 本地命令 → 直接执行（/clear, /status 等）
    │
    └── AI 消息 → DCCBridgeManager.send_message()
                    │
                    ▼ (独立线程)
              bridge_core.py (OpenClawBridge)
                    │ WebSocket RPC
                    ▼
              OpenClaw Gateway ──► AI Agent (LLM) 处理
                    │
              ┌─────┴────────────────────┐
              │                          │
         需要调用工具?              纯文本回复
              │                          │
              ▼                          ▼
         mcp-bridge 插件          流式 delta/final 事件
         (Gateway 内)                    │
              │ MCP JSON-RPC             │
              ▼                          │
         MCP Server (Maya内 :8081)       │
              │                          │
              ▼                          │
         command_queue.put()             │
              │                          │
              ▼ (主线程消费)             │
         adapter.execute_on_main_thread()│
              │                          │
              ▼                          │
         DCC 原生 API 执行               │
              │                          │
              ▼                          │
         结果 → Gateway → AI → 继续推理  │
              │                          │
              └──────────┬───────────────┘
                         │
                         ▼
              bridge_core.py 收到响应
                         │ Qt signal
                         ▼
              DCCBridgeManager.signals.ai_message / response_complete
                         │
                         ▼
              chat_panel.py 更新消息气泡
```

### 5.2 AI 流式响应（Qt signal 驱动）

```
OpenClaw Gateway
    │ WebSocket delta 事件
    ▼
bridge_core.py (独立 asyncio 线程)
    │ on_ai_message 回调
    ▼
bridge_dcc.py
    │ Qt signal: ai_message.emit(state, text)
    ▼
chat_panel.py (Qt 主线程, signal/slot 自动线程切换)
    │
    ▼
消息气泡逐字追加显示
```

**与 UE 的关键区别**：UE 通过文件轮询（Python 写 JSON → C++ FTSTicker 读取），DCC 通过 Qt signal（Python 直接通知 Python UI），延迟更低、代码更简洁。

### 6.1 必需依赖

### 6.2 安装策略

```
插件首次启动
    │
    ▼

    │

    │
    └── 无 → 后台线程 pip install --target Lib/
              │
              ▼
         安装完成 → 加载模块 → 正常启动
```

与 UEClawBridge 策略一致：安装到插件私有 `Lib/` 目录，不污染 DCC 环境。

## 七、与 UEClawBridge 的差异对照

| 方面 | UEClawBridge | DCCClawBridge |
|------|-------------|---------------|
| 语言 | C++ + Python | 纯 Python |
| UI 框架 | Slate (UE 原生) | PySide2/Qt |
| 主线程调度 | `FTSTicker` + 文件轮询 | `QTimer` + `executeDeferred` |
| AI 响应回传 | 文件轮询（Python 写文件 → C++ 读） | Qt signal/slot（直接推送） |
| 连接状态同步 | `_bridge_status.json` 文件 | Qt signal `connection_changed` |
| 生命周期 | `UEditorSubsystem` (C++) | `userSetup.py` (Python) |
| 部署方式 | UE 插件 (.uplugin) | Python 包（复制到 scripts/） |
| MCP 工具数量 | 1 个（`run_ue_python`） | 1 个（`run_python`） |
| 其余功能 | Python API + OpenClaw Skill | Python API + OpenClaw Skill |
| 万能执行器 | `run_ue_python` (操作 `unreal` 模块) | `run_python` (操作 `maya.cmds` / `pymxs`) |
| 上下文变量 | S=选中Actor, W=World, L=unreal | S=选中对象, W=当前场景, L=cmds/pymxs |
| 平台通信适配 | `openclaw_bridge.py` (13KB) | `bridge_dcc.py` (6KB) |
| 共享核心 | `bridge_core.py` (20KB) | `bridge_core.py` (20KB) — 同一份 |
